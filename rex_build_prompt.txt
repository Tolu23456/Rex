§0  WHAT REX IS AND WHAT IT MUST BECOME
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Rex is a statically typed, compiled programming language that targets x86-64
Linux (ELF64) via hand-authored NASM assembly. It is not built on LLVM, GCC,
Cranelift, or any external backend. Every instruction in every Rex binary is
deliberately placed by the Rex compiler. Rex does not delegate optimisation to
another tool — it owns every clock cycle end-to-end.

The existing compiler is written in NASM assembly itself: six object files, one
linked binary, 37 passing tests. Your job is to take this foundation and build
Rex into a language that is complete, safe, beautiful, and measurably faster
than C in its target domains.

─────────────────────────────────────────────────────────────────────────────
0.1  THE PERFORMANCE THESIS
─────────────────────────────────────────────────────────────────────────────

Rex does not aim to match C. Rex aims to surpass C in real-world programs.
This is achievable — not because Rex is magic, but because C compilers face
structural disadvantages that Rex does not:

  WHY C IS NOT OPTIMAL:
  ─────────────────────
  1. C's aliasing rules force GCC/Clang to be conservative. Any pointer could
     alias any other pointer, so the optimizer must assume memory effects at
     every store. Rex has no pointers in the safe subset — it has typed
     collections with known layout. The compiler knows exactly what is aliased
     and what is not.

  2. C's generic malloc/free is a stop-the-world general purpose allocator with
     lock contention, metadata overhead, and fragmentation. Rex programs using
     `use mm arena:` bypass malloc entirely — they bump-allocate from a pre-mapped
     slab. Arena allocation is 20-100x faster than malloc for short-lived objects.

  3. C auto-vectorisation is a compiler guess. If the loop doesn't look like
     what GCC expects, it won't vectorise. Rex's `each` and `blast` loops ALWAYS
     vectorise because the compiler generates the SIMD instructions directly —
     there is no "vectoriser" that might fail.

  4. C string operations call glibc. glibc's memcpy/memset/strlen are excellent,
     but they are generic. Rex generates string operations specialised to the
     exact element size, alignment, and data layout of each call site.

  5. C hash maps (glib, unordered_map) carry metadata, separate key/value pools,
     and dynamic dispatch. Rex dict is open-addressing, inline key storage, and
     one lookup is one hash + one or two cache-line accesses.

  6. C compilers spend billions of cycles in LLVM or GCC's optimisation passes.
     Rex emits already-optimal NASM. Time spent compiling is time not running.

  WHAT THIS MEANS IN PRACTICE:
  ─────────────────────────────
  - Bulk collection operations (sum, filter, map, sort on large seqs):
    Rex should be 1.5x–3x faster than equivalent C with glibc malloc.
  - String-heavy workloads (split, join, search on many short strings):
    Rex SSO + SIMD search should be 2x–5x faster than C with char* strings.
  - Allocation-heavy workloads (building graphs, parsers, trees):
    Rex arena allocation should be 10x–50x faster than C with malloc.
  - Arithmetic-heavy workloads (integer/float loops):
    Rex should match handwritten NASM within 2%.
  - Startup time:
    Rex binaries have no dynamic linker, no .init_array, no C runtime.
    A "Hello, Rex!" binary starts and exits in under 50 microseconds.

  WHAT "SURPASS C" DOES NOT MEAN:
  ─────────────────────────────────
  It does not mean beating a C expert with perfect hand-tuned malloc, custom
  allocators, and explicit SIMD intrinsics. It means beating typical C programs
  as written by a skilled C programmer using the standard toolchain (gcc -O3
  with glibc). That is the realistic comparison point.

─────────────────────────────────────────────────────────────────────────────
0.2  CORE DESIGN PRINCIPLES (in priority order)
─────────────────────────────────────────────────────────────────────────────

  1. Correctness    — Wrong output is worse than no output. Never sacrifice
                      correctness for speed. A fast wrong answer is a bug.

  2. Speed          — Rex programs must be faster than equivalent C in
                      collection-heavy, allocation-heavy, and string-heavy
                      workloads. Assembly-parity in arithmetic workloads.

  3. Safety         — No silent undefined behaviour anywhere in the safe
                      subset. Every out-of-bounds, overflow, and null access
                      must produce a clear, located error message.

  4. Expressiveness — The language must feel natural. If a feature requires
                      excessive ceremony to use, the design is wrong.

  5. Minimalism     — The compiler binary must be small. The language surface
                      must be learnable in a weekend. No bloat. No cruft.

  6. Transparency   — A programmer should be able to predict, from the source
                      code alone, what NASM will be emitted. No hidden costs,
                      no invisible allocations, no surprise GC pauses unless
                      explicitly opted into.

§1  ABSOLUTE RULES — THE AI MUST FOLLOW THESE WITHOUT EXCEPTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

These rules are not guidelines. Violating any of them is a project failure.

RULE 1:  NO STUBS. EVER.
  If a feature cannot be fully implemented right now, leave its syn.md marker
  as 🔧 or 📋 and move on. Do not write a function that compiles but:
    - Returns 0 or null silently
    - Crashes on valid input
    - Drops data without a documented reason
    - Prints nothing when output is expected
  A broken partial implementation is worse than no implementation because it
  makes the test suite lie. "It passes" is meaningless if the test is wrong.

RULE 2:  STATUS MARKERS ARE SACRED.
  ✅ = Fully implemented, fully tested, passes all 5 test categories (§9).
  🔧 = Token lexed. Parser/codegen not started.
  📋 = Not started at all.
  Flip a marker to ✅ only when ALL of the following are true:
    (a) The feature produces correct output for all cases in syn.md examples.
    (b) At least 5 tests in tests/ cover it (T1–T5 from §9).
    (c) The benchmark in bench/ exists and shows the target performance.
    (d) The syn.md example code blocks compile clean under `rex check`.
  If you flip a marker early, you are lying to the next person who reads the doc.

RULE 3:  EVERY RUNTIME ERROR IS A DIAGNOSTIC, NOT A CRASH.
  An acceptable runtime error message has FOUR parts:
    1. File name: the .rex source file where the error occurred
    2. Line number: the exact line, 1-indexed
    3. Error category: bounds / overflow / null / key / arena / type / etc.
    4. Context: the actual bad value and what was expected
  Examples of REQUIRED format:
    "rex: myapp.rex:47: bounds error: index 12 out of range [0..7]"
    "rex: parser.rex:103: dict key not found: \"author\""
    "rex: alloc.rex:12: arena overflow: need 8192 bytes, 6 remain in slab"
    "rex: compute.rex:88: arithmetic overflow: int add wrapped at 9223372036854775807"
    "rex: tree.rex:201: null dereference: seq 'children' was never initialised"
  Bare "segfault", "panic", or silent termination are NEVER acceptable.
  Every runtime error call site must include the source location — embed the
  line number as a compile-time immediate in the error-path NASM.

RULE 4:  NO UNDEFINED BEHAVIOUR IN SAFE CODE.
  Every operation in the safe subset (no #unsafe) must have a fully specified,
  documented outcome. If an operation COULD invoke undefined behaviour, insert
  a runtime check. If that check is too expensive, design around the operation.
  Rex does not have C's "if you do X, anything can happen" — Rex has "if you do
  X, you get this specific error message and the program exits cleanly".
  The ONLY exception is code explicitly inside `#unsafe` blocks, where the
  programmer has accepted full responsibility by annotation.

RULE 5:  FIVE TESTS PER FEATURE, NO EXCEPTIONS.
  For every feature you implement, write test files in tests/ covering:
    T1: Three distinct normal cases with typical inputs
    T2: One boundary case (empty, single element, zero, max value, etc.)
    T3: One large-scale case (N > 1000) to exercise growth and realloc paths
    T4: One error case that SHOULD produce a specific runtime error message;
        verify the message text by capturing stderr in the test harness
    T5: One composition case using this feature alongside 3+ other features
        in a realistic mini-program (e.g., build a word-frequency dict from
        a string, filter by count, sort keys, output formatted results)
  Tests that do not cover all five categories are incomplete. Add them.

RULE 6:  SURGICAL MODIFICATIONS ONLY.
  The existing 37-test-passing codebase is the floor. You may only modify
  existing code when:
    (a) A new feature genuinely requires changing an existing code path, AND
    (b) You have a written plan for exactly what changes and why, AND
    (c) All 37 existing tests still pass after the change.
  If you break existing tests, STOP and fix them before proceeding. Do not
  "fix forward" by changing the expected output of existing tests.

RULE 7:  PERFORMANCE TARGETS ARE HARD CONSTRAINTS.
  Rex must surpass C. The following are MINIMUM acceptable results measured
  against gcc -O2 with glibc on a Linux x86-64 machine (Intel or AMD):
    - Integer arithmetic loops:         Rex ≥ 98% of raw NASM speed
    - seq push/pop amortised:           Rex ≥ 110% of C vector<int> speed
    - seq sort (introsort) N=1M:        Rex ≥ 105% of qsort() speed
    - str contains (SIMD) N=10KB:       Rex ≥ 200% of strstr() speed
    - str split/join N=1000 strings:    Rex ≥ 150% of equivalent C with malloc
    - dict get/set 1M operations:       Rex ≥ 120% of C uthash speed
    - Arena allocation 10M allocs:      Rex ≥ 1000% of malloc speed
    - Process startup ("Hello, Rex!"):  Rex ≤ 50 microseconds total
    - Binary size ("Hello, Rex!"):      Rex ≤ 4 KB ELF binary
  If a benchmark does not meet its target, you must either:
    (a) Fix the implementation until it does, OR
    (b) Document exactly why it is architecturally impossible and propose a
        design change. "It's close enough" is not a valid response.

RULE 8:  THE COMPILER IS DETERMINISTIC.
  Given the same source file, the Rex compiler ALWAYS produces byte-for-byte
  identical output. This means:
    - No timestamps embedded in the binary (not in comments, not in data)
    - No random label suffixes (use a monotonically incrementing counter seeded
      at 0 for each compilation run, never from rdrand or /dev/urandom)
    - No ASLR-influenced output (do not store addresses during compilation)
    - No filesystem-order-dependent module loading (sort imports alphabetically)
    - No uninitialized memory reads in the compiler itself (zero all arenas)
  This enables binary reproducibility, diffing, and caching.

RULE 9:  THE COMPILER DOES NOT LEAK.
  The compiler process uses a single monolithic arena for all compilation data.
  This arena is allocated once at startup and freed once at shutdown. Every
  allocation during compilation goes through this arena. Nothing uses malloc
  directly. After compilation completes, valgrind --leak-check=full must print
  "0 bytes in 0 blocks".

RULE 10: IMPLEMENTATION AND DOCUMENTATION MUST BE IN SYNC.
  After implementing any feature, update syn.md before committing. The spec
  and the code must NEVER diverge. If implementation reveals a flaw in the
  spec, correct the spec with a "Design revision:" note explaining the change
  and its rationale. Then commit both the code and the updated spec together.

RULE 11: ONE FEATURE, ONE COMMIT.
  Each commit implements exactly one syn.md feature. The commit message format:
    "feat(section): implement <feature name> [✅]"
  Example: "feat(2.3.5): implement each loop with prefetch and mutating form [✅]"
  No batched commits. No "misc fixes". The git log is a feature map.

RULE 12: NO UNCONTROLLED GLOBAL STATE.
  The compiler has a designated set of BSS globals (var_table, label counter,
  ALLOC_CTX_PTR, etc.). New global state must go through an approved mechanism:
    - Compilation-lifetime data: the compilation arena
    - Runtime data in generated programs: declared BSS in the emitted NASM
    - Temporary pipeline state: passed on the call stack
  Do not add new BSS globals to the compiler itself without architectural review.

RULE 13: EVERY FEATURE MUST HAVE A BENCHMARK.
  The bench/ directory contains one benchmark file per major feature group.
  Each benchmark:
    - Runs for at least 2 seconds of wall time (loop until time >= 2s)
    - Measures nanoseconds per operation using RDTSC or clock_gettime(CLOCK_MONOTONIC)
    - Prints: "feature_name: N ns/op (M ops in T seconds)"
    - Has a matching bench/ref/<feature>.c that does the equivalent operation in C
    - The comparison is automated: `rex bench` runs both and prints the ratio
  The bench harness is part of the Rex toolchain. It is not optional.

RULE 14: ERROR RECOVERY IN THE COMPILER.
  The compiler must not abort on the first error. It must:
    - Report every error it finds in a single run
    - Continue parsing after an error using sync points (next statement, next prot)
    - Cap error reporting at 20 errors (after that: "too many errors, stopping")
    - Never crash with a segfault or SIGILL in the compiler itself for any input
  A compiler that crashes on malformed input is a security vulnerability and an
  unusable tool.

RULE 15: ASSEMBLY QUALITY.
  The NASM emitted by Rex must be legible. Each emitted block must begin with
  a comment naming the Rex source construct that generated it:
    ; --- for loop: i in 0..100 ---
    mov r12, 0          ; i = 0
    .for_1_top:
    cmp r12, 100
    jge .for_1_end
    ...
    inc r12
    jmp .for_1_top
    .for_1_end:
  This makes it possible to read the compiler output, debug code generation,
  and verify that the emitted code is correct. Comments do not affect performance.

2.1  TYPE SYSTEM
─────────────────────────────────────────────────────────────────────────────

2.1.1  Type inference (📋)
  The compiler's type inference must be local (Hindley-Milner-lite, no global
  unification). Inference works on the RHS of declarations and protocol returns.

  Rules:
    - `x = 5`        → int  (integer literal)
    - `x = 3.14`     → float  (float literal)
    - `x = "hello"`  → str  (string literal)
    - `x = true`     → bool
    - `x = 'R'`      → char
    - `x = @add(a,b)` → return type of @add (looked up from protocol table)
    - `x = a + b`    → type of a if a and b are the same type; promote to float
                        if one is float
    - `x = seq_val[i]` → element type T of seq[T]
    - `x = dict_val[key]` → value type T of dict[T]

  Inference does NOT cross scope boundaries. If `x` is used before its
  declaration is reached, that is an error even with inference.

  The `:` mutation rule still applies to inferred variables. The sigil is on
  write sites, not declarations. `x = 5; :x = 6` works exactly as before.

  Explicit types always win when present. `int x = 5` — no inference needed.
  `x = 5` — infer int. Conflict between declared and inferred type is a
  compile-time error with a clear message: "rex: type conflict: declared int
  but initialiser has type float at file.rex:12"

  Implementation note: during semantic analysis, unresolved type slots are
  filled by examining the initialiser expression. A second pass resolves any
  remaining slots. If a slot is still unresolved after two passes, it is an
  error: "rex: cannot infer type of 'x': provide an explicit type annotation"

2.1.2  `tup[T...]` (📋)
  Tuples are fixed-size, heterogeneous, stack-allocated. Zero header overhead.
  Zero dynamic dispatch. Zero heap allocation.

  Storage: each field occupies sizeof(T) bytes on the stack, aligned to its
  natural alignment. The tuple struct is the concatenation of field layouts with
  padding for alignment. The total size is rounded to 8 bytes.

  Field access `.0`, `.1` etc. are compile-time constant offsets from the
  tuple's base address. No runtime computation needed.

  Codegen for `int a, str b, float c = record`:
    mov rax, [rbp - tuple_offset + 0]    ; field .0 (int)
    mov rbx, [rbp - tuple_offset + 8]    ; field .1 (str pointer)
    movsd xmm0, [rbp - tuple_offset + 16] ; field .2 (float)

  Tuple return: protocols returning `(int, int)` pack into rax (first) and
  rdx (second). Protocols returning 3+ values: caller allocates a stack buffer
  and passes its address as the last argument in rdi before call; callee writes
  fields at [rdi + 0], [rdi + 8], [rdi + 16], etc.

  Partial destructuring with `_` skips the load — the offset is simply not read.
  This is a compile-time optimisation, not a runtime no-op.

  `.len()` on a tuple is always `N` where N is the compile-time field count.
  Replace with the integer literal directly — no runtime call.

2.1.3  `set[T]` (📋)
  Hash set backed by the same SipHash-2-4 used for dict. The set is an open-
  addressing table of T values (not key-value pairs). T must be a primitive
  type or str for hashing to be defined.

  Layout: `[capacity: 8][length: 8][table: cap * sizeof(T)]`
  A slot is "empty" when it contains the zero value for T. This means T=int
  sets cannot hold the value 0 — use `dict[bool]` if 0 is a valid member.
  (Document this limitation clearly in syn.md.)

  API:
    `.add(val)`         — insert val; no-op if already present
    `.remove(val)`      — remove val; no-op if absent
    `.has(val)`         — bool membership test, O(1) average
    `.len()`            — current element count
    `.clear()`          — remove all; keep allocation
    `.is_empty()`       — len() == 0
    `.union(other)`     — new set containing all elements of both
    `.intersect(other)` — new set containing elements in both
    `.diff(other)`      — new set: self minus other
    `.is_subset(other)` — every element of self is in other
    `.is_superset(other)` — every element of other is in self
    `.each(fn)`         — call fn(val) for every member
    `.to_seq()`         — seq[T] of all members (order unspecified)

  Literal syntax: `{1, 2, 3}` (no colons → set, not dict).
  Empty set literal: `{}` is ambiguous — use `set[int]()` or similar.
  Parser: at `{`, look ahead for `:` after first value → dict; no `:` → set.

2.1.4  `const` and `volatile` (📋)
  `const int LIMIT = 128`
    - Compile-time constant. Any mutation attempt (`:LIMIT = ...`) is a
      compile-time error: "rex: cannot mutate const 'LIMIT'"
    - The value is eligible for constant folding: any expression that depends
      only on compile-time constants is evaluated at compile time and replaced
      with its result in the emitted NASM.
    - `const` variables do not occupy a memory slot — they are inlined as
      immediates wherever they are used.

  `volatile int tick`
    - Disable register caching for `tick`. Every read of `tick` must emit a
      memory load; every write must emit a memory store. No register may be
      considered "holding the value of tick" across any instruction boundary.
    - Use case: memory-mapped I/O addresses, signal-modified variables.
    - Implementation: in register allocation, mark the variable's slot as
      "never resident in register". Every access emits `[rbp-N]` directly.

2.2  OPERATORS — FULL SPECIFICATION
─────────────────────────────────────────────────────────────────────────────

2.2.1  `not` operator (🔧 → must reach codegen)
  Parsing: `not expr` — unary prefix, lowest precedence
  Codegen for bool operand:
    Evaluate expr → rax (0, 1, or 2 for false/true/unknown)
    `cmp rax, 2; je .not_unknown` — preserve unknown
    `xor rax, 1`                  — flip 0↔1
    `.not_unknown:`               — unknown stays 2
  Codegen for int operand: `not rax` (bitwise complement, all bits flipped)
  Kleene logic: not(unknown) = unknown. not(false) = true. not(true) = false.
  Type check: `not` only applies to bool or int. Apply to float → compile error.
  Composition: `not a and b` parses as `(not a) and b` — NOT binds tightest.

2.2.2  `is` / `is not` (📋)
  Semantics: value identity, not structural equality. For primitives (int, float,
  bool, char, byte), `is` is equivalent to `==`. The distinction matters for
  future reference types (Stage 10) where `is` checks pointer identity and `==`
  checks structural equality. Implement now to reserve the syntax.
  Codegen: `cmp rax, rbx; sete al; movzx rax, al` → 0 or 1 in rax
  `is not`: `setne al` instead of `sete al`
  `ptr is not null`: `test rax, rax; setne al; movzx rax, al`
  Result type: always bool (never unknown — identity is not probabilistic).

2.2.3  Membership `in` (📋)
  `val in seq[T]`:
    Emit a linear scan loop. The compiler knows T's size, so index arithmetic
    is `[base + i * sizeof(T)]`. For int (8 bytes): `cmp [base + i*8], val`.
    Use SSE2 `pcmpeqq` to check 2 int64 values per iteration when N > 8.
    Returns bool. Short-circuits on first match.

  `key in dict[T]`:
    Compute SipHash-2-4 of key string. Probe table at `(hash % cap)` slot.
    If slot is empty → false. If slot has matching key → true. Probe linearly
    on collision. This is a full hash-lookup, not a scan.

  `sub in str`:
    Use the SSE2 string search (§4.3.2). First byte broadcast, scan 16 bytes
    per iteration, confirm full match on hit. Returns bool.

  `val in set[T]`:
    Identical to dict lookup — hash val, probe the set table.

  Codegen: `in` produces 0 or 1 in rax. Used in `if` conditions as a bool.

2.2.4  Pipeline operator `->` (📋)
  `expr -> @fn()` is sugar for `@fn(expr)` — expr is inserted as first argument.
  `@a() -> @b() -> output` chains: output(@b(@a()))
  Multi-arg: `@fn(extra_arg) -> @process()` does not apply here — `->` always
  passes the result as the FIRST (and only new) argument. Existing args in
  the target call are passed as written; the piped value is prepended.
  Example: `nums -> nums.filter(fn(int x) -> bool: x > 0)` — filter gets nums
  as the implicit receiver, not an argument. Pipeline does not change method
  call semantics; it only works with `@protocol` calls and built-in keywords.
  Type checking: the return type of the LHS must exactly match the first
  parameter type of the RHS. Mismatch → compile error with types shown.

2.2.5  Syscall intercept `$` (📋)
  Only in `#unsafe` blocks. Outside `#unsafe`: compile error, exact message:
    "rex: file.rex:N: '$' requires #unsafe annotation on enclosing protocol"
  Codegen: `$(rax_val, rdi_val, rsi_val, rdx_val, r10_val, r8_val, r9_val)`
    mov rax, rax_val
    mov rdi, rdi_val    ; if provided
    mov rsi, rsi_val    ; if provided
    mov rdx, rdx_val    ; if provided
    mov r10, r10_val    ; if provided
    mov r8,  r8_val     ; if provided
    mov r9,  r9_val     ; if provided
    syscall
  Return value is in rax. Assign with: `int result = $(60, 0)` — the assignment
  captures rax after syscall.
  Args may be variables, literals, or expressions. The compiler evaluates each
  arg into a temp register and moves to the ABI position register.

2.2.6  `flip` / `rand` (📋)
  `flip b` where b is bool:
    `xor byte [b_stack_addr], 1` — in-place bit flip, no load-store round-trip.
    For unknown (value 2): `xor` would give 3, which is invalid. Check first:
    `cmp byte [b_stack_addr], 2; je .flip_skip` — do not flip unknown.
    `.flip_skip:` — leave unknown as unknown.

  `rand`:
    `.retry: rdrand rax; jnc .retry` — RDRAND returns 0 in CF on failure.
    Result is a full 64-bit random integer in rax. Assign to int variable.

  `unknown` bool source:
    `rdrand rax; jnc .retry` — get a random bit
    `and rax, 1` — isolate bit 0
    Assign to bool variable (0 = false, 1 = true). Not 2/unknown — unknown
    comes from explicit `:b = unknown` or from Kleene logic producing unknown.

2.2.7  Hardware flags `carry` / `overflow` (📋)
  These are volatile reads — they are only valid IMMEDIATELY after an arithmetic
  instruction. The compiler must ensure no intervening instruction between the
  arithmetic and the flag read.

  Pattern:
    ; compute a + b into rax
    add rax, rbx
    ; read carry flag — NEXT INSTRUCTION MUST BE SETC
    setc al
    movzx rax, al     ; bool: 0 or 1
    ; read overflow flag from a separate add — need to redo the add
    ; (cannot use both carry and overflow from the same operation easily
    ;  unless storing both immediately after)

  Implementation constraint: `carry` and `overflow` reads are only valid
  directly after an arithmetic expression statement. Emitting them mid-expression
  or after control flow is a compile-time error:
    "rex: file.rex:N: 'carry' must appear as the next statement after arithmetic"

2.2.8  String repetition `*` (📋)
  `s * n` where s is str and n is int:
    Compile-time: if n == 0 → empty string. If n == 1 → return s unchanged.
    Runtime: allocate `s.len() * n` bytes. Use `rep movsq` to fill each copy.
    Return new str. Source s is unchanged.
  n must be non-negative. n < 0 → compile-time error if literal; runtime check
  if variable: "rex: file.rex:N: string repetition count -3 is negative"

─────────────────────────────────────────────────────────────────────────────
2.3  CONTROL FLOW — FULL SPECIFICATION
─────────────────────────────────────────────────────────────────────────────

2.3.1  `match` structural pattern matching (📋)
  `match x:` where x is any expression.
  Each `arm_type: body` arm is a type match. The first matching arm executes.

  Dense integer arms (e.g., `is 1:`, `is 2:`, ..., `is N:` with no gaps > 3):
    Build a jump table:
      `.jt: dq .arm_0, .arm_1, .arm_2, ...` in .rodata
    Codegen:
      `cmp rax, 0; jl .else_arm`   ; below range
      `cmp rax, N; jg .else_arm`   ; above range
      `jmp [.jt + rax*8]`          ; O(1) dispatch

  Sparse integer arms: binary search over sorted case values. Each comparison
  is a `cmp` + conditional jump. Binary search is O(log n) comparisons.

  Type match (`int:`, `float:`, `str:`): compare `typeof x` to each type token.
  `else:` arm: the fallthrough after all others are tried.
  No fallthrough between arms — each arm ends with `jmp .match_end`.
  Exhaustiveness: if matching a bool (3 values) or byte (256 values), warn on
  non-exhaustive match. For other types, exhaustiveness is not checked.

2.3.2  `repeat N:` counted loop (📋)
  N must be a compile-time integer constant (literal or const variable).
  N = 0 → the entire `repeat` block is dead code; emit nothing.
  N = 1 → emit the body once with no loop structure.
  N > 1 → emit the dec/jnz loop:

    ; --- repeat 8: ---
    mov rcx, 8
    .repeat_1_top:
        <body>
    dec rcx
    jnz .repeat_1_top
    .repeat_1_end:

  `rcx` is the loop counter. It is NOT visible inside the body as a variable.
  `stop` inside `repeat` patches a `jmp .repeat_1_end` at the stop site.
  `skip` inside `repeat` patches a `jmp .repeat_1_top` at the skip site.
  Nested `repeat`: the outer `rcx` is saved with `push rcx` before the inner
  loop and restored with `pop rcx` after. If 3 levels deep, spill to stack slot.

2.3.3  `stop N` / `skip N` multi-level (📋)
  The break-patch stack (already used for single-level stop/skip) is extended
  with a depth counter per entry. Entry format:
    [jmp_target_slot: 8 bytes][depth: 4 bytes][type: 4 bytes (break/continue)]
  `stop N`: walk back N entries, patch ALL intermediate loops with their exit
    labels (so they clean up), then jump to the Nth outer loop's exit.
  `skip N`: jump to the Nth outer loop's condition-check / top label.
  `stop 1` and `skip 1` are identical to bare `stop` and `skip`.
  `stop N` where N exceeds current loop depth: compile-time error with count.

2.3.4  Loop `else:` (📋)
  Requires one `bool` scratch slot per loop that has an `else:` clause.
  Before loop: `mov byte [loop_N_broke], 0`
  At every `stop` site: `mov byte [loop_N_broke], 1` before the JMP
  After loop exit label: `cmp byte [loop_N_broke], 0; jne .loop_N_skip_else`
  Else block body follows. `.loop_N_skip_else:` after else block.
  The scratch slot is reclaimed at scope exit (same mechanism as for-loop vars).

2.3.5  `each` collection iterator (🔧 → full implementation)
  Works on: seq[T], arr[T,N], str (yields char), dict[T] (yields k,v), set[T]

  For seq[T] — element-only form:
    ; --- each item in nums: ---
    mov r12, [nums_ptr + 8]      ; r12 = len
    mov r13, [nums_ptr + 16]     ; r13 = data pointer
    xor r14, r14                 ; r14 = index = 0
    .each_1_top:
    cmp r14, r12
    jge .each_1_end
    prefetcht0 [r13 + r14*8 + 64]   ; prefetch next cache line
    mov rax, [r13 + r14*8]          ; rax = current element
    ; --- body ---
    inc r14
    jmp .each_1_top
    .each_1_end:

  For str — yields char:
    Same structure but: `movzx rax, byte [r13 + r14]` (no *8)

  For dict[T] — yields k, v:
    Iterate the hash table slots. Skip empty and tombstone slots.
    Yield key pointer (str) in rax, value in rbx.

  With index `each i, item in col:`:
    Expose r14 as `i` inside the body. r14 is read-only — attempting to write
    it via `:i = ...` is a compile-time error.

  Mutating form `each :item in nums:`:
    Instead of loading into rax and discarding, keep the computed address:
    `lea rax, [r13 + r14*8]` — rax holds the address of the element.
    `:item = expr` becomes `mov [rax], computed_expr_result`.
    The `:` on the element name is part of the `each` syntax, not a statement.

  `stop` and `skip 1` inside each: work identically to `for`/`while`.
  `stop N` / `skip N`: see §2.3.3 — each counts as one loop level.

2.3.6  `try` / `except` / `finally` (📋)
  Implementation strategy: setjmp/longjmp style using a Rex-internal mechanism
  (no libc setjmp — roll it from scratch using `rsp` and `rbp` saves).

  Rex maintains a per-scope "exception frame" slot in BSS:
    `REX_EXC_FRAME  resq  64`   ; room for 64 nested try levels

  On `try:` entry:
    - Save rsp, rbp, and the address of the `except:` block into the frame
    - Increment the frame depth counter

  When `err("msg")` fires INSIDE a try block:
    - Instead of calling sys_exit(1): check if frame depth > 0
    - If yes: write the message string to `REX_EXC_MSG` slot
    - Restore rsp and rbp from the frame, jump to the `except:` block address
    - Decrement frame depth

  `except msg:`:
    - Load the string from `REX_EXC_MSG` into the `msg` variable
    - Execute the except block

  `finally:`:
    - Emitted AFTER the try block and AFTER the except block (if present)
    - Control reaches it in both the normal and error paths
    - Implemented with two copies of the finally body (one for normal path,
      one for error path) OR with a jmp-to-shared pattern — use jmp-to-shared
      to avoid code bloat. Emit the finally body once; both normal and error
      paths jump to it; a flag distinguishes which path ran.

  `warn` inside try: NEVER intercepted. warn always reaches stderr.
  Nested try: each level increments the frame depth. Inner except catches first.
  `err` inside `except`: propagates to the next outer try frame.

─────────────────────────────────────────────────────────────────────────────
2.4  SEQUENCES — FULL METHOD SET AND ALGORITHMIC REQUIREMENTS
─────────────────────────────────────────────────────────────────────────────

The seq[T] header is `[capacity: 8 bytes][length: 8 bytes][data: N*sizeof(T)]`.
All seq methods return the SAME pointer if operating in-place, or a NEW pointer
for functional methods (map, filter, etc.). Callers are responsible for
assigning the returned pointer.

GROWTH STRATEGY: doubling from initial capacity 8.
  When push would overflow: new_cap = max(old_cap * 2, 8).
  New buffer = alloc(new_cap * sizeof(T)). Copy old data. Free old buffer.
  Amortised O(1) push. The growth path is cold — mark with cold JMP.

SMALL SEQ OPTIMISATION: For N ≤ 4 elements where T is a primitive, store
inline in a 48-byte stack struct to avoid heap allocation entirely. The flag
bit in capacity distinguishes heap from inline storage.

METHODS — ALGORITHMIC SPECIFICATIONS:

  `.sort()` — Introsort (hybrid quicksort + heapsort + insertion sort):
    - N ≤ 16: use insertion sort directly (excellent cache behaviour, no recursion)
    - N > 16: start quicksort with median-of-three pivot selection
    - Recursion depth > 2*floor(log2(N)): switch to heapsort (O(n log n) guarantee)
    - Partition: three-way partition (Dutch National Flag) for arrays with many
      duplicate elements — this is O(n) when all elements are equal
    - Introsort guarantees O(n log n) worst case and O(n log n) average case.
      It is the algorithm used by C++ std::sort.
    - In-place. No extra allocation for the sort itself.

  `.sort_by(fn)` — Same introsort skeleton. The comparator fn is called via
    `call rax` (register-indirect). Cache fn pointer in a callee-save register
    across the entire sort to avoid reloading from the stack every comparison.

  `.sort_desc()` — Introsort with reversed pivot selection and comparison.
    Do NOT call sort() then reverse() — that is needlessly two passes.

  `.unique()`:
    - N ≤ 64: allocate a sorted copy, scan for adjacent duplicates, write to
      output seq. O(n log n) time. Original seq order is preserved by tracking
      first-occurrence indices.
    - N > 64: one-pass using the set[T] implementation. Hash each element,
      skip if already seen, output if new. O(n) time, O(n) space. Preserves
      original order exactly.
    ALWAYS preserves original order — this is documented behaviour.

  `.reduce(fn, init)` — Left fold. Keep accumulator in a callee-save register
    (r12) for the entire reduction. fn is called via `call rax` (register-
    indirect with fn pointer in a callee-save register to avoid reloading).
    No intermediate allocation. Output is the final value of the accumulator.

  `.zip(other)` — Allocate `min(len_a, len_b)` tup[T,U] elements. Iterate
    both seqs in lockstep. Each iteration writes two values side by side.
    The output type is `seq[tup[T,U]]` — requires tup to be implemented first.

  `.flatten()` — Input: seq[seq[T]]. Output: seq[T].
    Pass 1: iterate inner seqs, accumulate total_len.
    Single allocation: data = alloc(total_len * sizeof(T)).
    Pass 2: memcpy each inner seq's data into the output buffer.
    Two passes, one allocation. Not N allocations.

  `.map(fn)` — Allocate output seq of the SAME length as input (known upfront).
    No reallocation during map. Fill with fn(element) for each element.
    The output element type U may differ from T — output is seq[U].

  `.filter(fn)` — Cannot know output length upfront. Start with capacity = len/2.
    Double when full (same growth strategy as push). Return trimmed allocation.

  `.each(fn)` — Tight register loop. fn pointer in r12. No allocation.
    Emit: `call r12` for each element.

  `.count(fn)` — Same loop as each. Accumulate counter in r13.

  `.find(fn)` — Return first element where fn returns true. Error if none found.
    Error message: "rex: file.rex:N: find(): no matching element in seq"

  `.find_index(fn)` — Return index (-1 if not found). No error on miss.

  `.index_of(val)` — Linear scan with SSE2 acceleration for int/float:
    Use pcmpeqq to compare 2 elements per iteration. For other types: scalar.

  `.sort_desc()` — See above.

  `.insert(i, val)` — Bounds check i. Shift elements right from i to end.
    Use memmove pattern (rep movsq backwards). O(n) is correct; no SIMD needed.

  `.swap_at(i, j)` — Load element i into temp, store element j at i, store
    temp at j. Three memory ops for int. For str pointers: same pattern.

  `.copy()` — Allocate new seq of same capacity. memcpy header + data.

─────────────────────────────────────────────────────────────────────────────
2.5  STRINGS — FULL METHOD SET AND ALGORITHMIC REQUIREMENTS
─────────────────────────────────────────────────────────────────────────────

STORAGE: heap header `[capacity: 8][length: 8][data: variable]`.

SMALL STRING OPTIMISATION (SSO):
  Strings ≤ 15 bytes are stored inline without heap allocation:
    [flag=SSO | len: 8 bytes][data[0..14]: 15 bytes][null: 1 byte]
  The flag bit is bit 63 of the first 8-byte word. If set: SSO. If clear: heap.
  All str methods must branch on this flag and handle both cases.
  SSO strings live on the stack and are copied when passed (value semantics).
  Heap strings carry their capacity and grow on concatenation.
  SSO → heap promotion: when a method would produce a result > 15 bytes,
  allocate a new heap string instead of extending the SSO buffer.

METHODS — ALGORITHMIC SPECIFICATIONS:

  `.upper()` / `.lower()` — ASCII-only (0x41-0x5A, 0x61-0x7A).
    SSE2 implementation processing 16 bytes per iteration:
    ; For lower() — convert uppercase to lowercase:
    movdqu xmm0, [src + i]      ; load 16 bytes
    movdqa xmm1, xmm0
    pcmpgtb xmm1, [CONST_40]    ; xmm1: 0xFF where byte > 'A'-1
    movdqa xmm2, xmm0
    pcmpgtb xmm2, [CONST_5A]    ; xmm2: 0xFF where byte > 'Z'
    pandn xmm2, xmm1            ; xmm2: 0xFF where 'A' <= byte <= 'Z'
    pand xmm2, [CONST_20]       ; xmm2: 0x20 where uppercase
    por xmm0, xmm2              ; set bit 5 → lowercase
    movdqu [dst + i], xmm0      ; store result
    Non-ASCII bytes (> 0x7F) pass through unchanged.
    Handle tail (len % 16 bytes) with scalar fallback.

  `.contains(sub)`, `.index_of(sub)`, `.last_index_of(sub)`:
    Two-Way String Matching algorithm. Pre-computation on the needle: O(m).
    Search phase: O(n + m) worst case. O(1) extra space.
    Additionally: use SSE2 to scan for the first byte of needle before invoking
    Two-Way. The SSE2 first-byte scan finds candidate positions 16x faster.
    Combined: first-byte SSE2 scan + Two-Way verification on hits.

    .last_index_of: reverse both haystack and needle before searching,
    then invert the returned index.

  `.replace(old, new)` / `.replace_first(old, new)`:
    Pass 1: find all occurrence positions (using Two-Way), count them → `cnt`
    Compute output length: `len - cnt * len(old) + cnt * len(new)`
    Single allocation of output length. Pass 2: fill segments and replacements.
    `.replace_first`: stop after first occurrence found.

  `.reverse()`:
    N ≤ 16: scalar swap loop.
    N > 16: SSE2 pshufb-based 16-byte lane reversal.
      Load 16 bytes, apply pshufb with [15,14,...,1,0] mask, store reversed.
      Handle middle for odd-sized strings. Join reversed lanes.
    Note: byte-level reversal only. UTF-8 codepoints spanning multiple bytes
    will be corrupted — document this explicitly. Users who need Unicode-correct
    reversal must call `.chars()` first.

  `.join(parts)`:
    Pass 1: sum all string lengths in parts + (parts.len-1) * sep.len
    Single allocation. Pass 2: memcpy each string, insert separator between.
    Never append incrementally — one allocation, one fill.

  `.split(sep)`:
    `sep = ""`: allocate seq[str] of len(s) elements, each a 1-char SSO string.
    `sep != ""`: scan haystack with Two-Way for sep, collect split points.
    Pre-count splits on first scan. Allocate seq[str] of count+1 elements.
    Second scan: fill each segment. Two passes, one seq allocation.

  `.repeat(n)`:
    n = 0 → empty string. n = 1 → return s unchanged.
    Allocate `len * n` bytes. Use `rep movsq` to copy len bytes, advancing
    destination by len each time. Efficient for long strings.

  `.chars()` → seq[char]:
    Allocate seq header + len elements. Each byte → char (single movzx per byte).

  `.bytes()` → seq[byte]:
    Allocate seq header + len elements. memcpy the raw data.

  `.pad_left(n, c)`:
    If len >= n: return copy of s unchanged.
    Else: allocate n bytes. Fill [0..n-len-1] with c (rep stosb). Copy s into
    [n-len..n-1]. Result is left-padded.

  `.count(sub)` — Two-Way scan, count non-overlapping matches.
  `.count(fn)` — char predicate loop, accumulate count.

─────────────────────────────────────────────────────────────────────────────
2.6  DICTIONARIES — FULL METHOD SET AND HASH MAP DESIGN
─────────────────────────────────────────────────────────────────────────────

LAYOUT: `[capacity: 8][length: 8][seed: 8][table: cap * slot_size]`
Each slot: `[key_ptr: 8][value: sizeof(T)][hash: 8]` — caching the hash avoids
rehashing on lookup if key_ptr is set. Empty slot: key_ptr = 0. Tombstone:
key_ptr = 1 (a sentinel, never a valid pointer).

HASH FUNCTION: SipHash-2-4 with per-dict random seed (from rdrand at init).
LOAD FACTOR: grow when len * 100 / cap > 75. Shrink when len * 100 / cap < 20.
GROWTH: double capacity; rehash all live entries (skip tombstones).
SHRINK: halve capacity; rehash all live entries. Minimum capacity: 8.

TOMBSTONE COMPACTION: at shrink time, tombstones are simply dropped — they are
not copied into the new table. After compaction, tombstone count = 0.

LOOKUP OPTIMISATION: because each slot stores the hash, a quick filter is:
  `cmp [slot + key_offset + 8], computed_hash; jne .next_slot`
This eliminates the full string comparison for most non-matching slots.
Only when hashes match do we invoke the full `strcmp` equivalent.

METHODS:
  `.entries()` — blocked until tup is implemented. Yields `seq[tup[str, T]]`.
  `.map(fn)` — fn signature: `fn(str key, T val) -> U`. Returns dict[U].
    Allocate output dict with same initial capacity. Iterate live slots.
  `.filter(fn)` — fn: `fn(str key, T val) -> bool`. Returns dict[T].
    Allocate output dict, insert matching entries.
  `.invert()` — T must be str (compile-time check). Swap key and value.
    Duplicate values → last key wins (documented). Allocate new dict[str].
  `.update(other)` — Iterate other's slots, set each in self.
    Overwrites self's values on key conflict.
  `.find_key(val)` — Linear scan over values using `memcmp`. Returns "" if none.
  `.has_value(val)` — Same linear scan, returns bool.
  `.each(fn)` — fn: `fn(str key, T val)`. Called for every live slot.

─────────────────────────────────────────────────────────────────────────────
2.7  OUTPUT, I/O, AND STRING INTERPOLATION
─────────────────────────────────────────────────────────────────────────────

2.7.1  String interpolation `{expr}` (📋)
  All Rex string literals support interpolation at parse time. The lexer scans
  string content for `{` (not `{{`) and tokenizes them as:
    TOK_STR_LITERAL "prefix"
    TOK_INTERP_OPEN
    ... tokens for the expression ...
    TOK_INTERP_CLOSE
    TOK_STR_LITERAL "suffix"
  The parser builds an INTERP_STR AST node containing an ordered list of
  (literal, expr) pairs.

  Codegen for `"x is {x} and y is {y}"`:
    1. Emit the literal prefix "x is " (pointer + length in .rodata)
    2. Evaluate x → rax; call rex_str_from_int to convert to temp str
    3. Emit the literal " and y is "
    4. Evaluate y → rax; call rex_str_from_int to convert
    5. Join all segments using a stack-allocated buffer sized to the sum of all
       lengths (computed first). One write per segment.
    6. The result is a temp str used immediately for output; freed after use.

  `{{` in source → literal `{` in output (lexer replacement, not runtime).
  `}}` → literal `}`.
  Nested `{` inside `{}` → compile error: "nested interpolation not allowed"
  `{@fib(10)}` → the `@` prefix is recognised inside interpolation context.
  `{val:spec}` → see §2.7.2.

2.7.2  Format specifiers `{val:spec}` (📋)
  The lexer recognises `:` after a non-whitespace expression inside `{}` as
  beginning a format specifier. The specifier is consumed as a raw token.

  Supported specifiers (all must be implemented):
    `.Nf`  — float, N decimal places. N is 0-15. Use Grisu2 + truncation.
    `Nd`   — integer, minimum width N, right-aligned, space-padded.
    `0Nd`  — integer, minimum width N, zero-padded. N is 1-20.
    `x`    — integer as lowercase hex. No prefix.
    `X`    — integer as uppercase hex. No prefix.
    `b`    — integer as binary. No prefix.
    `o`    — integer as octal. No prefix.
    `Ns`   — string, minimum width N, left-padded with spaces (right-aligned).
    `+d`   — integer with explicit `+` sign for positive values.
    `e`    — float in scientific notation: 3.14e+00.
    `g`    — float: use `.Nf` if exponent is in [-4, precision], else `e`.

  Float-to-decimal: implement Grisu2 (Florian Loitsch, 2010). This algorithm
  produces the shortest decimal representation that round-trips through
  double parsing. Do NOT call sprintf, snprintf, or any libc formatting.
  Grisu2 is a fixed-point algorithm using 64-bit integer arithmetic that runs
  in O(1) time for typical floats. Reference implementation is in the Grisu2
  paper. Implement it in NASM as a standalone subroutine.

2.7.3  `fmt(template)` (📋)
  Identical codegen to string interpolation but the result is a heap-allocated
  str returned in rax, not written to stdout. The caller receives a str they
  can store, pass, or output later.

2.7.4  `show(x)` (📋)
  Same as `output(x)` but does NOT emit a trailing newline. Implementation:
  the output subroutines all share a "write_newline" flag parameter. show calls
  them with the flag cleared.

2.7.5  `flush()` (📋)
  Rex writes directly via sys_write (fd 1). There is no user-space stdout buffer.
  `flush()` therefore emits `sys_fsync(1)` — this forces the kernel to drain
  any kernel-side pipe buffers. In most programs this is unnecessary but it is
  semantically correct. Emit: `mov rax, 74; mov rdi, 1; syscall` (sys_fsync).

2.7.6  `debug(x)` (📋)
  Writes to fd 2 (stderr). Format: `<type>: <value>\n`
  For each type: compute the type name string (compile-time constant), then
  the value representation using the same formatting as output, then write
  both to fd 2 separated by ": ".
  Example output: `seq[int]: [1, 2, 3]`
  `debug(x)` is a statement, not an expression. It produces no value.

2.7.7  `write(buf)` (📋)
  `buf` must be `seq[byte]` or `arr[byte, N]`.
  Codegen: `mov rax, 1; mov rdi, 1; mov rsi, [buf+16]; mov rdx, [buf+8]; syscall`
  No newline. No conversion. Raw bytes to stdout.
  Verify buf is not null before syscall.

2.7.8  `input("prompt")` (📋)
  Write prompt string to fd 1 (no newline — `show` semantics).
  Allocate initial read buffer: 256 bytes on the heap.
  `sys_read(0, buf, capacity)` into the buffer.
  Scan for `\n`. If found before capacity: create str of len (excluding `\n`).
  If not found: double the buffer, re-read from where we left off, repeat.
  Return the str. The `\n` is NOT included in the returned string.
  EOF (read returns 0): return empty string `""`.

2.7.9  `warn("msg")` (📋)
  Write "rex warning: " + msg + "\n" to fd 2. Never exits. Interpolation works.
  Codegen: identical to `err()` except no `sys_exit(1)` at the end.

─────────────────────────────────────────────────────────────────────────────
2.8  MEMORY ALLOCATOR CONTEXTS — FULL SPECIFICATION
─────────────────────────────────────────────────────────────────────────────

2.8.1  Allocator Vtable Design
  Every allocator is a pointer to a vtable:
    struct alloc_vtable {
      u64 alloc_fn;   ; fn(size) -> ptr
      u64 free_fn;    ; fn(ptr, size)
      u64 reset_fn;   ; fn() — bulk reset
      u64 ctx_data;   ; pointer to allocator-specific state
    }
  The current vtable pointer lives at `ALLOC_CTX_PTR = 0x447000` (BSS).
  All allocation inside any Rex runtime call routes through:
    mov rax, [ALLOC_CTX_PTR]     ; vtable ptr
    mov rdi, size
    call [rax]                    ; call alloc_fn

  On `use mm X:` entry:
    push qword [ALLOC_CTX_PTR]   ; save old vtable ptr onto the shadow stack
    mov [ALLOC_CTX_PTR], new_vtable_ptr

  On `use mm X:` exit:
    pop qword [ALLOC_CTX_PTR]    ; restore old vtable ptr

  The shadow stack is a fixed BSS array `ALLOC_CTX_STACK resq 64` with a depth
  counter. Nesting deeper than 64 is a runtime error.

2.8.2  Arena Allocator (✅ — verify and harden)
  State: `[base: 8][bump: 8][end: 8]` — mmap'd contiguous slab.
  alloc_fn: `if bump + size > end: err("arena overflow")`
             `else: ptr = bump; bump += (size + 7) & ~7; return ptr`
  Alignment: all allocations are 8-byte aligned (round up with mask).
  free_fn: no-op (arena doesn't free individual allocations).
  reset_fn: `bump = base` — O(1), no syscall.
  On scope exit: `munmap(base, end - base)`.

2.8.3  Pool Allocator (✅ — verify and harden)
  State: `[block_size: 8][free_list: 8][slab_list: 8]`
  Each slab is an mmap'd block of `slab_size = 256 * block_size` bytes.
  alloc_fn: pop from free list if non-empty. Else mmap a new slab, push all
    new blocks onto free list, then pop.
  free_fn: push ptr back onto free list.
  reset_fn: for each slab, rebuild the free list with all blocks.

2.8.4  Stack Allocator (📋)
  Uses the hardware stack directly. No mmap. No data structure.
  alloc_fn: `sub rsp, size; return rsp` — emitted as inline NASM, not a call.
  free_fn: no-op at individual call sites. On scope exit: `add rsp, total_allocated`.
  Constraint: allocations must be freed in LIFO order (which scope exit guarantees).
  Warning: if the compiler estimates total stack use > 1 MB, emit a warning at
  compile time. The estimate sums max simultaneous live allocations.

2.8.5  Heap Allocator (📋)
  Uses Rex's own implementation of a best-fit free list allocator.
  Do NOT call malloc(). Implement a simple header-based heap:
    - Initial heap: mmap 4 MB.
    - Each block: `[size: 8][flags: 8][data...]` where flags bit 0 = in-use.
    - alloc_fn: scan free list for best fit. Split if oversized. Coalesce on free.
    - free_fn: mark as free, coalesce adjacent free blocks.
  This is not the fastest allocator but it is correct and self-contained.
  The arena and pool allocators are preferred for performance-critical code.

2.8.6  Static Allocator (📋)
  Allocations survive all scope exits. Never freed.
  Implementation: mmap a fixed-size region at startup (16 MB default).
  Bump-allocate from it. No free. No reset. On overflow: mmap more.
  Use for: global configuration, interned strings, protocol tables.

2.8.7  Named Contexts `ctx` (📋)
  `ctx name = arena(N)`:
    Allocate the arena vtable and state in the compilation arena (compiler-side).
    At runtime: emit `mmap(0, N, PROT_RW, MAP_ANON|MAP_PRIVATE, -1, 0)` once
    at the declaration site. Store in a BSS slot named for `name`.
  `reset name`: emit `mov [name_bump], [name_base]` — one instruction.
  `free name`: emit `munmap([name_base], name_size)`.
  Named ctx used as allocator: `use mm name:` — installs the ctx's vtable.

2.8.8  GC Modes — Full Specification (📋)
  `use gc sweep:`:
    At scope entry: record the current var_table watermark.
    At scope exit: walk the var_table from watermark to current top.
    For each entry that is a heap pointer (seq, dict, str): check if it is
    reachable from any stack variable. If not: free it.
    This is a simple mark-and-sweep over the scope's variables.

  `use gc ref:`:
    Each heap allocation gets a reference count word prepended to the block:
    `[refcount: 8][... data ...]`
    On assignment `:a = b` where a was a heap pointer: decrement a's refcount,
    free if 0. Increment b's refcount.
    On scope exit: decrement refcount of all scope-local heap variables.

  `use gc gen:`:
    Maintain two regions: young (16 MB) and old (64 MB).
    All new allocations go into young. When young is 75% full: minor collection.
    Minor collection: mark reachable objects in young, copy survivors to old.
    When old is 75% full: major collection (full sweep of both regions).
    This mirrors the design of the JVM's G1 collector at a much simpler scale.

  `use gc inc:`:
    Spread sweep work over time. After every N bytes allocated, do M bytes of
    sweep work where M = allocated_since_last_gc / target_heap_multiplier.
    Prevents long pauses at the cost of constant small overhead.

  `use gc region:`:
    All allocations in the scope go into one mmap'd region. On scope exit:
    one munmap frees everything. No per-object tracking. Identical to arena mm
    but at the GC abstraction level. The fastest possible GC: no GC at all.

2.8.9  User-Defined Allocators and GCs (📋)
  `mm myalloc: alloc(size): <body> free(ptr): <body> reset: <body>`
    Compiles each method as a protocol. Builds a vtable in BSS with pointers
    to the three compiled protocols. Exposes the name `myalloc` as a `use mm`
    option.

  `gc mygc: mark(ptr): <body> collect: <body> trigger: <body>`
    Same pattern for GC vtables.

  `use keyword slab as mm myalloc`:
    Register `slab` as an alias for `myalloc` in the allocator keyword table.
    Parse-time binding — not runtime. Collision with built-ins → compile error.

─────────────────────────────────────────────────────────────────────────────
2.9  PROTOCOLS — ADVANCED FEATURES
─────────────────────────────────────────────────────────────────────────────

2.9.1  `#memo` (✅ — harden to support all parameter type combinations)
  Cache key: concatenate all parameter values as a byte string.
    int → 8 bytes little-endian
    float → 8 bytes IEEE 754
    str → length (4 bytes) + content bytes
    bool → 1 byte
    char → 1 byte
    byte → 1 byte
  Cache storage: a global dict[int] keyed by the byte-string (or a faster
  purpose-built hash structure for performance).
  Cache miss → run body, store (key → return value), return value.
  Cache hit → return stored value without entering body.
  Memory: the memo cache is never cleared. For recursive protocols this is
  correct (memoized Fibonacci is the canonical case). For protocols with many
  unique inputs this can grow unboundedly — document this.
  Thread safety: not required (Rex is single-threaded).

2.9.2  `#pure` (📋)
  Compile-time annotation. The compiler verifies:
    - No calls to output, show, err, warn, debug, write, flush, input
    - No `:var = ...` writes to variables declared outside the protocol scope
    - No `use mm` or `use gc` blocks (allocator context changes are a side effect)
  If any violation is found: compile error.
  Effect on optimisation:
    - If a `#pure` call result is not used: elide the call entirely
    - Adjacent calls to the same `#pure` protocol with the same arguments:
      compute once, reuse the result (call-site memoization)
    - `#pure` protocols may be reordered relative to other `#pure` calls

2.9.3  `#total` (📋)
  Programmer asserts the protocol terminates for all inputs. No runtime check.
  Compiler effects:
    - Enable loop unrolling for any known-bounds loops in the body
    - `#total #pure` combination: eligible for compile-time evaluation if all
      arguments are compile-time constants (constant folding across protocols)

2.9.4  `#hot` (📋)
  Align the protocol's entry label to 64 bytes (one cache line):
    `align 64` before the protocol label in emitted NASM.
  Emit the protocol FIRST in its section (before any other protocols in the
  same source file) so it lands in the first instruction cache page.
  The assembler/linker controls final placement, but the source ordering hint
  is effective for most ELF linkers.
  `#hot` does NOT imply `#inline`. They are orthogonal.

2.9.5  `#cold` (📋)
  Emit the protocol body in a `.cold` section separate from `.text`.
  Jumps TO the cold protocol use the `0x2E` branch-not-taken hint prefix.
  Effect: the cold protocol is placed far from hot code in the final ELF,
  reducing instruction cache pressure.

2.9.6  `#safe` / `#unsafe` (📋)
  Default: all protocols are `#safe`.
  `#safe` effects: at the semantic analysis stage, scan the body for:
    - `$` (syscall intercept) → error if found
    - Direct pointer arithmetic (loading from a raw integer address) → error
    - `carry` / `overflow` flag reads outside the immediately-following-arithmetic
      context → error
  `#unsafe` removes all three restrictions. The decorator is opt-in.
  Nested call: calling an `#unsafe` protocol from a `#safe` protocol is a
  compile-time error. You must annotate the caller as `#unsafe` too.

2.9.7  Multiple return values (📋)
  Up to 2: first in rax, second in rdx. SysV ABI allows this for integer-class
  types. Float returns use xmm0 and xmm1.
  3+ returns: caller allocates return buffer on the stack:
    `sub rsp, N*8`         ; room for N return values
    `mov [call_ret_ptr], rsp`  ; pass buffer address as implicit last arg
  The callee writes return values at [ret_ptr], [ret_ptr+8], [ret_ptr+16], etc.
  Destructuring: `int a, int b, int c = @triple()` generates the buffer,
  then loads [rsp+0], [rsp+8], [rsp+16] into a, b, c respectively.

2.9.8  Lambda `fn` (📋)
  Non-capturing lambda → function pointer. Zero overhead vs a protocol call.
  Capturing lambda → closure. Layout:
    [fn_ptr: 8][captured_var_1: sizeof][captured_var_2: sizeof][...]
  Closure allocation: in the active mm context (or stack if `use mm stack:`).
  Calling a closure: the caller passes the closure pointer; the callee loads
  the function pointer from [closure+0], the captured vars from [closure+8+].
  Type: `prot(T, U -> V)`. Stored in a typed variable. Called via `@name(args)`.
  Lambda equality: two lambdas are equal only if they are the same object —
  there is no structural equality for protocol types.

2.9.9  Arbitrary parameter count (currently capped at 6)
  Extend to unlimited parameters using stack-passing for args 7+.
  Args 1-6: rdi, rsi, rdx, rcx, r8, r9 (as now).
  Args 7+: pushed right-to-left before the call, cleaned by caller after.
  Codegen for calling `@fn(a, b, c, d, e, f, g, h)`:
    mov rdi, a
    mov rsi, b
    mov rdx, c
    mov rcx, d
    mov r8, e
    mov r9, f
    push h          ; arg 8 pushed first (rightmost first)
    push g          ; arg 7 pushed second
    call _fn
    add rsp, 16     ; caller cleans: 2 args * 8 bytes
  Callee accesses arg 7 at [rbp+16], arg 8 at [rbp+24].
  Rex programmers do not see this complexity — it is transparent.

2.10  MODULES AND STANDARD LIBRARY
─────────────────────────────────────────────────────────────────────────────

2.10.1  Module system (📋)
  Resolution order:
    1. Current directory: `./math.rex`
    2. Project modules: `./modules/math.rex`
    3. Standard library: `STDLIB_PATH/math.rex` (env var or default `/usr/lib/rex/`)
    4. Compile error: "module 'math' not found in search path"

  Circular import detection: build the import graph (DAG). If a cycle is
  detected: emit all nodes in the cycle in the error:
    "rex: circular import detected: a → b → c → a"

  Module compilation: each imported module is compiled independently first,
  then its protocol table (name → address) is made available to the importer.
  Link order is determined by topological sort of the import DAG.

  `import math` — makes `@math.sqrt(x)` available.
  `from math import sqrt` — makes `@sqrt(x)` available.
  `from math import sqrt as sq` — makes `@sq(x)` available.
  `from math import *` — wildcard import. Use sparingly. Document it in syn.md.

2.10.2  `math` standard library (📋)
  All math functions implemented in NASM without libc:
    sqrt(float) → float:  `sqrtsd xmm0, xmm0`  (SSE2 — hardware instruction)
    pow(float, float):    `log` + `exp` via minimax polynomial
    log(float):           natural log via identity: log(m*2^e) = log(m) + e*log(2)
                          Compute log(m) for m in [1, 2) with degree-7 minimax poly
    log2(float):          `log(x) / log(2)` using the above
    log10(float):         `log(x) / log(10)` using the above
    exp(float):           Horner's method degree-8 polynomial approximation
    sin(float):           x87 `fsin` instruction OR SSE minimax poly for speed
    cos(float):           x87 `fcos` OR minimax poly
    tan(float):           x87 `fptan` OR minimax poly
    floor(float):         SSE4.1 `roundsd xmm0, xmm0, 1` or x87 fallback
    ceil(float):          SSE4.1 `roundsd xmm0, xmm0, 2` or x87 fallback
    round(float):         SSE4.1 `roundsd xmm0, xmm0, 0` or x87 fallback
    abs(float):           `andpd xmm0, [SIGN_MASK]` (clear sign bit)
    min(float, float):    `minsd xmm0, xmm1`
    max(float, float):    `maxsd xmm0, xmm1`
    pi:                   compile-time constant in .rodata (64-bit IEEE 754)
    e:                    compile-time constant in .rodata (64-bit IEEE 754)

  Precision target: all transcendental functions must be accurate to ≤ 1 ULP
  (unit of least precision) relative to the IEEE 754 double result.
  Reference: test against mpfr for 10000 random inputs before marking ✅.

2.10.3  `io` standard library (📋)
  All implemented as Rex protocols over raw syscalls.
    read_file(str path) → str:
      open(path, O_RDONLY) → fd
      fstat(fd) → size
      allocate str of size bytes
      read(fd, ptr, size)
      close(fd)
      return str

    write_file(str path, str content):
      open(path, O_WRONLY|O_CREAT|O_TRUNC, 0644) → fd
      write(fd, content.data, content.len)
      close(fd)

    append_file(str path, str content):
      open(path, O_WRONLY|O_CREAT|O_APPEND, 0644) → fd
      write(fd, content.data, content.len)
      close(fd)

    read_lines(str path) → seq[str]:
      read_file(path) → content
      content.lines()   ; split on \n

2.10.4  `os` standard library (📋)
    args() → seq[str]: read argc and argv from the stack at program entry.
      Rex's _start must save argc and argv to BSS slots. args() returns a
      seq[str] built from them.
    env(str key) → str: scan environ for "KEY=VALUE", return "VALUE" or "".
    exit(int code): sys_exit(code).
    cwd() → str: sys_getcwd into a 4096-byte buffer, return as str.
    time_ns() → int: clock_gettime(CLOCK_MONOTONIC) → 64-bit nanoseconds.

2.10.5  `json` standard library (📋)
    parse(str s) → dict[str]:
      Hand-written recursive descent JSON parser in Rex.
      Supports: object, array (as seq[str] with nested serialisation), string,
      number (stored as str), true/false/null (stored as str).
      Nested objects: values that are themselves objects are stored as
      serialised JSON strings (simplification for V5.0).
      On invalid JSON: err("json: invalid token at position N: ...")

    stringify(dict[str] d) → str:
      Emit `{`, iterate keys in sorted order, emit `"key": "value"` pairs,
      emit `}`. Sort for deterministic output.
      String escaping: emit `\"` for double quotes inside values.

2.10.6  `net` standard library (📋)
    tcp_connect(str host, int port) → int: sys_socket + sys_connect. Returns fd.
    tcp_listen(int port) → int: sys_socket + sys_bind + sys_listen. Returns fd.
    tcp_accept(int server_fd) → int: sys_accept. Returns client fd.
    send(int fd, str data) → int: sys_write(fd, data). Returns bytes written.
    recv(int fd, int max_bytes) → str: sys_read(fd, buf, max_bytes). Returns str.
    close(int fd): sys_close.
    Error handling: all syscalls check return value. Negative → err() with errno.

2.10.7  `str_utils` standard library (📋)
    encode_base64(str s) → str
    decode_base64(str s) → str
    encode_url(str s) → str
    decode_url(str s) → str
    parse_int(str s, int base) → int  (base 2-36)
    parse_float(str s) → float

§3  COMPILER ARCHITECTURE — REQUIRED STRUCTURE AND PIPELINE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The Rex compiler is itself a performance-critical program. It must compile a
1000-line Rex file in under 100ms. The pipeline has six stages:

  Source → Lexer → Parser → Semantic Analysis → Codegen → Optimizer → Emit

Each stage is a separate .asm source file. Data flows between stages through
arena-allocated data structures. No stage retains state after it returns.

─────────────────────────────────────────────────────────────────────────────
3.1  LEXER
─────────────────────────────────────────────────────────────────────────────

Single-pass, byte-by-byte scan. One byte of lookahead only.

Token structure (20 bytes, cache-aligned):
  [type: 4 bytes][line: 4 bytes][start: 8 bytes][len: 4 bytes][pad: 4 bytes]

Token type is a 32-bit integer enum. The full token table:
  TOK_INT_LIT, TOK_FLOAT_LIT, TOK_STR_LIT, TOK_CHAR_LIT, TOK_BOOL_LIT,
  TOK_IDENT, TOK_COLON, TOK_ASSIGN, TOK_PLUS, TOK_MINUS, TOK_MUL, TOK_DIV,
  TOK_MOD, TOK_AND, TOK_OR, TOK_NOT, TOK_EQ, TOK_NEQ, TOK_LT, TOK_GT,
  TOK_LE, TOK_GE, TOK_LPAREN, TOK_RPAREN, TOK_LBRACKET, TOK_RBRACKET,
  TOK_LBRACE, TOK_RBRACE, TOK_DOT, TOK_COMMA, TOK_ARROW, TOK_AT, TOK_HASH,
  TOK_DOLLAR, TOK_AMP, TOK_PIPE, TOK_XOR, TOK_TILDE, TOK_SHL, TOK_SHR,
  TOK_DOTDOT, TOK_NEWLINE, TOK_INDENT, TOK_DEDENT, TOK_EOF,
  TOK_KW_INT, TOK_KW_FLOAT, TOK_KW_STR, TOK_KW_BOOL, TOK_KW_CHAR, TOK_KW_BYTE,
  TOK_KW_SEQ, TOK_KW_ARR, TOK_KW_DICT, TOK_KW_SET, TOK_KW_TUP, TOK_KW_PROT,
  TOK_KW_IF, TOK_KW_ELIF, TOK_KW_ELSE, TOK_KW_FOR, TOK_KW_WHILE, TOK_KW_EACH,
  TOK_KW_REPEAT, TOK_KW_STOP, TOK_KW_SKIP, TOK_KW_PASS, TOK_KW_RETURN,
  TOK_KW_IN, TOK_KW_IS, TOK_KW_NOT, TOK_KW_AND, TOK_KW_OR, TOK_KW_WHEN,
  TOK_KW_MATCH, TOK_KW_TRY, TOK_KW_EXCEPT, TOK_KW_FINALLY, TOK_KW_USE,
  TOK_KW_MM, TOK_KW_GC, TOK_KW_CTX, TOK_KW_RESET, TOK_KW_FREE, TOK_KW_OWN,
  TOK_KW_MOVE, TOK_KW_ALIGN, TOK_KW_CONST, TOK_KW_VOLATILE, TOK_KW_IMPORT,
  TOK_KW_FROM, TOK_KW_AS, TOK_KW_FN, TOK_KW_STEP, TOK_KW_SWAP, TOK_KW_FLIP,
  TOK_KW_RAND, TOK_KW_CARRY, TOK_KW_OVERFLOW, TOK_KW_UNREACHABLE, TOK_KW_ASSERT,
  TOK_KW_BLAST, TOK_KW_PIPE, TOK_KW_OUTPUT, TOK_KW_SHOW, TOK_KW_ERR, TOK_KW_WARN,
  TOK_KW_DEBUG, TOK_KW_WRITE, TOK_KW_FLUSH, TOK_KW_INPUT, TOK_KW_FMT,
  TOK_KW_TRUE, TOK_KW_FALSE, TOK_KW_UNKNOWN, TOK_KW_NULL,
  TOK_INTERP_OPEN, TOK_INTERP_CLOSE,
  TOK_DOC_COMMENT, TOK_LINE_COMMENT (discarded)

Indentation: Rex uses indentation for blocks. The lexer must track indent level
and emit TOK_INDENT / TOK_DEDENT tokens. Use a stack of indent levels. Each
significant newline followed by a changed indent level emits the appropriate token.

String content is NOT copied at lex time. The token stores a pointer into the
source buffer (which is fully loaded into memory at the start of compilation)
and a length. No allocation during lexing. The source buffer lives for the
entire compilation — it is arena-allocated at startup.

Escape sequences handled during lex (convert in-place in the source buffer copy):
  \n → 0x0A, \t → 0x09, \r → 0x0D, \\ → 0x5C, \" → 0x22, \' → 0x27, \0 → 0x00
  \xNN → byte value NN (hex)
  Unknown escape → compile error: "rex: file.rex:N: unknown escape sequence '\\X'"

Block comments `/* */`: consumed by lexer as whitespace. Unclosed → error.
Doc comments `///`: emit TOK_DOC_COMMENT with the content. Consumed by parser
and attached to the next `prot` declaration.

─────────────────────────────────────────────────────────────────────────────
3.2  PARSER
─────────────────────────────────────────────────────────────────────────────

Recursive descent, hand-written in NASM. No parser generator. LL(1) grammar
with one token of lookahead. Two exceptions resolved by the existing lookahead:
  - `{expr}` vs `{key: val}` dict literal: peek at what follows the first value
  - `fn(...)` as expression vs statement: fn is always an expression

AST node structure. Every node is a tagged union:
  [tag: 4 bytes][type: 4 bytes][children: up to 4 child pointers (8 bytes each)]
  Some nodes have inline data (immediate integers, string pointers) in the
  child pointer slots.

All nodes are allocated from the compilation arena. No malloc in the parser.

Error recovery: on a syntax error, emit the error, then advance to the next
synchronisation point. Sync points: TOK_NEWLINE at indent level 0, TOK_KW_PROT,
TOK_EOF. Report ALL syntax errors, not just the first. Halt at 20 errors.

Grammar overview (not exhaustive — implement all Rex syntax from syn.md):
  program    := (import | prot_def | stmt)*
  prot_def   := decorator* 'prot' IDENT '(' params ')' ('->' type)? ':' block
  decorator  := '#' IDENT
  stmt       := assign | decl | output | control | loop | try | mm_block | expr
  decl       := type IDENT ('=' expr)? | IDENT '=' expr  ; inferred
  assign     := ':' IDENT ('[' expr ']')? '=' expr       ; mutation
  block      := INDENT stmt+ DEDENT
  expr       := or_expr ('->' call_expr)?
  or_expr    := and_expr ('or' and_expr)*
  and_expr   := not_expr ('and' not_expr)*
  not_expr   := 'not'? cmp_expr
  cmp_expr   := add_expr (cmp_op add_expr)?
  add_expr   := mul_expr (('+' | '-') mul_expr)*
  mul_expr   := unary (('*' | '/' | '%') unary)*
  unary      := ('-' | '~')? postfix
  postfix    := primary ('.' IDENT call_args? | '[' expr ']')*
  call_args  := '(' (expr (',' expr)*)? ')'
  primary    := IDENT | literal | '@' IDENT call_args | 'fn' fn_expr
              | '(' expr ')' | collection_literal

─────────────────────────────────────────────────────────────────────────────
3.3  SEMANTIC ANALYSIS
─────────────────────────────────────────────────────────────────────────────

Three passes:
  Pass 1 — Protocol hoisting: collect ALL top-level protocol names and their
    signatures. This enables forward references: protocol A may call protocol B
    that is defined after A in the source file.

  Pass 2 — Type checking and mutation analysis:
    - Walk the AST, resolve every identifier to its type.
    - Track which variables have `:var = ...` write sites (mutable) vs not.
    - Variables with no write sites: mark as compile-time constant (fold-eligible).
    - Check: no read of an uninitialized variable (no write before read in scope).
    - Check: type compatibility on every operation (see coercion rules below).
    - Check: `#pure` protocols — no I/O, no external writes.
    - Check: `#unsafe` scope for `$` and pointer operations.
    - Check: decorator compatibility (e.g., `#inline #noinline` is a conflict).
    - Resolve all `const` and `volatile` annotations.

  Pass 3 — Dead code and reachability:
    - Mark statements after `return`, `stop`, `err()` as unreachable.
    - Emit warnings for unreachable code (not errors — the code is just dead).
    - Mark `#cold` protocols and inline cold error paths.

  Coercion rules (all implicit, all happen at codegen time):
    int → float:    promote with `cvtsi2sd`
    char → int:     zero-extend with `movzx`
    byte → int:     zero-extend with `movzx`
    bool → int:     zero-extend with `movzx` (0, 1, or 2)
  All other conversions: require explicit cast function. Implicit coercion
  between incompatible types → compile error with types shown.

  Scope stack: a linked list of scope frames. Each frame has a table of
  (name → type, stack_offset, flags) entries. Lookup: scan current frame,
  then parent frames up to the protocol frame. Protocol boundary stops lookup
  (no closure-through-protocol unless `fn` captures).

─────────────────────────────────────────────────────────────────────────────
3.4  CODE GENERATION — DETAILED REQUIREMENTS
─────────────────────────────────────────────────────────────────────────────

Codegen emits NASM text into a string buffer (arena-allocated, grown as needed).
Each emitted block is preceded by a comment naming the Rex construct.

REGISTER ALLOCATION — LINEAR SCAN:
  Build live ranges for each variable (first use to last use, in instruction
  order). Sort intervals by start point. Allocate registers greedily. On spill:
  choose the interval with the furthest end point (minimises future spills).

  Register pool: rax, rbx, rcx, rdx, rsi, rdi, r8, r9, r10, r11, r12, r13, r14
  Callee-saved registers (per SysV ABI): rbx, rbp, r12, r13, r14, r15.
  These must be saved in the prologue and restored in the epilogue if used.
  Caller-saved (volatile): rax, rcx, rdx, rsi, rdi, r8, r9, r10, r11.

  Float pool: xmm0–xmm7 for float variables.
  Callee-saved floats: xmm8–xmm15 (SysV ABI).

INSTRUCTION SELECTION:
  For every Rex AST node, select the OPTIMAL instruction sequence. Key choices:
    Integer add: `add rax, rbx` (1 cycle latency on all modern CPUs)
    Integer mul by power of 2: `shl rax, N` (not imul — saves 2 cycles)
    Integer mul by 3: `lea rax, [rax + rax*2]` (1 cycle, not imul)
    Integer mul by 5: `lea rax, [rax + rax*4]`
    Integer mul by 9: `lea rax, [rax + rax*8]`
    Integer mul by other: `imul rax, N` (3 cycle latency — unavoidable)
    Integer div by power of 2: `sar rax, N` (signed, not idiv — saves ~20 cycles)
    Comparison result as int: `cmp; setcc al; movzx rax, al` (no branch needed)
    Ternary-style: use cmov where possible to avoid branches.

STACK FRAME LAYOUT:
  [rbp-8]: first local variable
  [rbp-16]: second local variable
  ... etc. (descending, 8-byte slots)
  [rbp+8]: return address
  [rbp+16]: first stack-passed argument (if any)
  Prologue: `push rbp; mov rbp, rsp; sub rsp, N` where N = total local var bytes
  N must be rounded to 16-byte boundary for SSE2 alignment requirement.
  Epilogue: `add rsp, N; pop rbp; ret` (or `leave; ret`).

LABEL NAMING CONVENTION:
  All labels use a monotonically incrementing suffix to guarantee uniqueness:
    `.for_001_top:`, `.for_001_end:`
    `.if_002_then:`, `.if_002_else:`, `.if_002_end:`
    `.prot_name:` — protocol entry labels use the protocol name directly
  This makes the output readable and predictable.

CALLING CONVENTION REQUIREMENTS:
  The compiler MUST save and restore rbx, r12, r13, r14, r15 in any protocol
  that uses them. The compiler tracks which callee-saved registers it has
  written and emits only the pushes and pops that are actually needed.
  Do NOT blindly push all callee-saved registers at every protocol entry.

─────────────────────────────────────────────────────────────────────────────
3.5  OPTIMIZER — FULL PASS SPECIFICATION
─────────────────────────────────────────────────────────────────────────────

The optimizer runs AFTER codegen, BEFORE writing NASM to disk. It operates
on the text buffer of emitted NASM as a sequence of structured instruction
records (parse the emitted text back into an instruction list first).

PASS 1: Constant folding and compile-time evaluation
  - Fold binary ops on two immediate integers: `add rax, 0` → eliminate.
    `mov rax, 5; add rax, 3` → `mov rax, 8`.
  - Fold unary ops: `mov rax, -5; neg rax` → `mov rax, 5`.
  - Fold comparisons: `mov rax, 10; cmp rax, 10; je .L` → `jmp .L`.
  - Fold `#pure #total` protocol calls where all args are compile-time constants:
    evaluate the protocol at compile time, replace the call with the result.
  - Fold `const` variable references: replace with the literal value.
  - Propagate known register values through a basic block (value numbering).
    If rax was just set to 5 and the next instruction is `cmp rax, 10`, emit
    `cmp rax, 10` with the known value tracked for further folding.

PASS 2: Strength reduction
  Every multiply that can be replaced with a cheaper instruction MUST be:
    imul rax, 2  → lea rax, [rax + rax]        ; latency 1 vs 3
    imul rax, 3  → lea rax, [rax + rax*2]       ; latency 1
    imul rax, 4  → shl rax, 2                   ; latency 1
    imul rax, 5  → lea rax, [rax + rax*4]       ; latency 1
    imul rax, 8  → shl rax, 3                   ; latency 1
    imul rax, 9  → lea rax, [rax + rax*8]       ; latency 1
    imul rax, N (power of 2) → shl rax, log2(N)
    idiv rax, N (power of 2) → sar rax, log2(N) ; latency 1 vs ~30

  Modulo by power of 2: `x % N where N = 2^k` → `and rax, N-1`

  Zero multiply: `imul rax, 0` → `xor rax, rax`
  Add 0: eliminate. Subtract 0: eliminate. Multiply by 1: eliminate.

PASS 3: Redundant instruction elimination
  Track register "def-use" chains within a basic block (instructions between
  two labels). Remove an instruction if:
    - Its output register is never read before being overwritten again.
    - It is a no-op: `mov rax, rax`, `add rax, 0`, `or rax, 0`.
    - It is a redundant move: `mov rax, rbx` followed immediately by
      `mov rbx, rax` with rbx not otherwise used → delete the second.
  Basic block boundary: any label or unconditional jump ends the block.

PASS 4: Jump chaining and elimination
  - `jmp .A` where `.A` begins with `jmp .B` → rewrite to `jmp .B`.
    Repeat until the target is not another unconditional jump.
  - `jmp .A` where `.A` is the next instruction → eliminate the jmp.
  - `cmp rax, rbx; je .A; jmp .B; .A:` → `cmp rax, rbx; jne .B; .A:`
    (invert condition and skip the unconditional jump).

PASS 5: Dead code elimination
  Mark reachable instructions starting from the function entry. Instructions
  not marked reachable are dead:
    - Instructions after unconditional `jmp` or `ret` before the next label.
    - Code only reachable from labels that are themselves unreachable.
  Delete dead instructions. Delete dead labels (labels with no references).

PASS 6: Loop-invariant code motion (LICM)
  For `for` loops: identify expressions whose operands are not modified inside
  the loop body. Hoist them before the loop:
    ; Before LICM:
    .loop_top:
      mov rax, [some_constant]    ; constant every iteration
      add rbx, rax
    ; After LICM:
      mov rax, [some_constant]    ; hoisted before loop
    .loop_top:
      add rbx, rax

  Only hoist expressions with no side effects (no I/O, no allocation).
  Only hoist when the expression is used every iteration (not inside a branch).

PASS 7: `#inline` expansion
  Collect all protocols decorated `#inline`. At every call site:
    - If the protocol body (after codegen) is ≤ 15 NASM instructions: inline it.
    - If recursive: do NOT inline (would cause infinite expansion).
    - If `#noinline`: do NOT inline even if `#inline` is also present (conflict
      resolved: `#noinline` wins; emit a compile warning about the conflict).
  Inlining: copy the body instructions, replace `ret` with a `jmp` to the
  after-call label, substitute parameter registers for argument evaluations.
  After inlining: re-run passes 1-5 over the inlined code.

PASS 8: Tail call optimisation (TCO)
  Detect tail-recursive protocol calls: a `call` immediately followed by `ret`
  (with no intervening operations or only a `mov rax, result`).
  Replace with: restore callee-save registers, set up arguments, `jmp` to the
  protocol's entry label. This converts stack-consuming recursion to a loop.
  Example: a recursive `@fib` call in the tail position becomes a loop.
  TCO applies to self-recursion only (same protocol). Mutual recursion TCO
  requires trampoline implementation — defer to V6.0.

PASS 9: Peephole post-processing
  Final micro-optimisations after all other passes:
    - `push rax; pop rbx` → `mov rbx, rax`
    - `mov rax, 0` → `xor rax, rax` (shorter encoding by 3 bytes)
    - `sub rax, 1` → `dec rax` (shorter encoding)
    - `add rax, 1` → `inc rax` (shorter encoding)
    - `cmp rax, 0` → `test rax, rax` (same flags, shorter or same encoding)
    - `jmp .A; .A:` → `.A:` (eliminate jump-to-next)
    - Multiple consecutive nops → collapse to one or zero

SUPERSCALAR SCHEDULING (Advanced — implement after all passes above are working):
  Modern x86 CPUs have multiple execution units. Instructions can execute in
  parallel if their dependencies allow it. The Rex optimizer should reorder
  independent instructions within a basic block to maximise throughput:
    - Separate a load from the instruction that uses its result by at least
      4 instructions (L1 cache hit latency on most Intel CPUs is 4 cycles).
    - Interleave integer and float operations when both are available — they
      use different execution units and do not compete.
    - Place `prefetcht0` instructions 64–128 bytes ahead of the data access
      they prefetch for.
  Implementation: build a dependency graph (DAG) for instructions in each
  basic block. Perform list scheduling: at each step, emit the ready
  instruction with the highest "schedule priority" (typically the one on the
  critical path to the last use).

§4  PERFORMANCE — SURPASSING C: COMPLETE TECHNICAL GUIDE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This section documents every technique used to make Rex programs faster than
their C equivalents. Every technique here must be implemented. Every benchmark
must validate the improvement.

─────────────────────────────────────────────────────────────────────────────
4.1  CPU FEATURE DETECTION AT STARTUP
─────────────────────────────────────────────────────────────────────────────

At Rex binary startup (before `main` equivalent), execute CPUID to detect:
  - SSE2: always present on x86-64. Required baseline.
  - SSE4.1: `roundsd` instruction (used for floor/ceil/round).
  - SSE4.2: `pcmpestri` (used for fast string search with character sets).
  - AVX: 256-bit registers. Enables 32-byte-per-iteration SIMD.
  - AVX2: `vpcmpeqb`, `vpgatherqq`, integer gather/scatter.
  - POPCNT: `popcnt` instruction (popcount for set[T] operations).
  - BMI2: `pext`, `pdep` (bit manipulation for hashing).
  - RDRAND: hardware random number generator (used for dict seeds and unknown).

Store results in a BSS flags word: `CPU_FLAGS resq 1`.
Rex runtime routines choose the best implementation path at runtime based on
this word. Use indirect dispatch: a table of function pointers where each slot
is filled with the best available implementation at startup.

Example for string search:
  If AVX2:    use 32-byte scan (32 bytes per iteration)
  If SSE4.2:  use `pcmpestri` with character-level semantics
  If SSE2:    use 16-byte `pcmpeqb` + `pmovmskb` scan
  Fallback:   scalar byte-by-byte (should never happen on x86-64)

This dispatch table is built in `rex_init` (called from `_start`) and costs
essentially nothing at runtime (one jmp through a pointer per first call;
branch prediction makes subsequent calls free).

─────────────────────────────────────────────────────────────────────────────
4.2  REGISTER DISCIPLINE — EXTENDED SPECIFICATION
─────────────────────────────────────────────────────────────────────────────

The single most impactful optimisation in assembly is keeping values in
registers instead of memory. Rex codegen must implement full linear-scan
register allocation as described in §3.4. Beyond that:

  HOT LOOP REGISTER ASSIGNMENTS:
  For any loop body that executes more than once, the compiler must ensure:
    - The loop counter is in a register for the entire loop duration.
    - The base pointer of any collection being iterated is in a register.
    - The length of any collection being checked against is in a register.
    - The function pointer for any lambda being called per-iteration is in r12.
  These are mandatory. A loop that reloads the base pointer from memory on
  every iteration has a guaranteed cache bottleneck.

  INSTRUCTION FUSION:
  Intel CPUs can fuse certain instruction pairs into one micro-op:
    `test rax, rax; jz .L` → fused compare-and-branch (1 micro-op)
    `cmp rax, 5; jle .L`   → fused compare-and-branch (1 micro-op)
    `dec rcx; jnz .top`    → fused decrement-and-branch (1 micro-op)
  The `dec + jnz` pattern for counted loops (`repeat N:`, range `for` loops)
  MUST use this pattern. It is the fastest possible loop control on Intel CPUs.

  ZERO IDIOM:
  `xor rax, rax` is preferred over `mov rax, 0` for two reasons:
    1. Shorter encoding (2 bytes vs 7 bytes for `mov rax, 0`).
    2. CPUs recognise it as a dependency-breaking zeroing idiom and execute
       it without waiting for the old value of rax.
  Always use `xor rax, rax` to zero a register.

─────────────────────────────────────────────────────────────────────────────
4.3  SIMD — COMPLETE IMPLEMENTATION REQUIREMENTS
─────────────────────────────────────────────────────────────────────────────

4.3.1  Memory copy and fill (used by join, repeat, extend, copy):
  N < 16 bytes: `rep movsb` or scalar `mov` instructions.
  16–127 bytes: `movdqu` (unaligned 16-byte load/store), loop unrolled 4x.
  128+ bytes: `movntdq` (non-temporal store, bypasses L1/L2 write cache)
    for data that will NOT be read again immediately (e.g., str.join output).
    For data that WILL be read immediately (e.g., seq.copy before use):
    use `movdqa` (aligned 16-byte) or `movdqu` (unaligned).
  AVX available: use `vmovdqu` (32-byte unaligned) for 2x throughput.
  The threshold to use `movntdq` vs `movdqa` is 256 bytes (below that, the
  cache-bypass overhead doesn't pay off for reads that follow soon after).

4.3.2  String search — the fastest possible implementation:
  Build: `rex_str_search(haystack_ptr, haystack_len, needle_ptr, needle_len)`
  Returns: index of first match, or -1.

  Step 1: If needle_len == 1: use `pcmpeqb` to find the single byte.
    Broadcast needle[0] to all 16 bytes of xmm1.
    Load 16 bytes of haystack at a time into xmm0.
    `pcmpeqb xmm0, xmm1` → all-ones for matching positions.
    `pmovmskb eax, xmm0` → bitmask of matches.
    `bsf eax, eax` → index of first match in this 16-byte chunk.
    This is 16x faster than a scalar byte loop.

  Step 2: If needle_len > 1: Two-Way algorithm with SSE2 first-byte filter.
    Use SSE2 to find positions where haystack[i] == needle[0] (filter).
    For each filter hit: verify full needle match with `cmpsb` / `repe cmpsb`.
    Two-Way preprocessing: compute the "period" and "critical position" of
    the needle in O(m) time. Then search in O(n) time.

  Step 3 (AVX2 available): use `_mm256_cmpeq_epi8` for 32-byte scan.
    Two loops of 32 bytes before dropping to 16 bytes for the tail.

4.3.3  Integer collection operations:
  `seq[int].sum()` for N > 8:
    Accumulate in two xmm registers (xmm0, xmm1) in parallel to hide latency:
    movdqu xmm0, [base + i*8]         ; elements 0-1
    movdqu xmm1, [base + i*8 + 16]    ; elements 2-3
    paddq xmm0, [base + i*8 + 32]     ; accumulate elements 4-5 into xmm0
    paddq xmm1, [base + i*8 + 48]     ; accumulate elements 6-7 into xmm1
    ; ... unroll 4x for best throughput
    paddq xmm0, xmm1                   ; combine accumulators
    movq rax, xmm0                     ; extract low 64 bits
    psrldq xmm0, 8                     ; shift high 64 bits
    movq rbx, xmm0
    add rax, rbx                       ; final sum

  `seq[int].min()` / `seq[int].max()`:
    Same structure as sum but using `pcmpgtq` + conditional select via masking.
    Process 2 elements per iteration with xmm0.

  `seq[int].contains(val)` for N > 8:
    Broadcast val to xmm1 (4 copies). Scan 16 bytes (2 elements) per iteration.
    `pcmpeqq xmm0, xmm1; pmovmskb eax, xmm0; test eax, eax; jnz .found`

4.3.4  Float operations with SSE2:
  All Rex float arithmetic MUST use SSE2 scalar double instructions.
  Never use x87 for float arithmetic (it has 80-bit internal precision that
  causes IEEE 754 round-trip failures and is slower on modern CPUs).

  `float.ceil()`, `.floor()`, `.round()`:
    IF SSE4.1 available: `roundsd xmm0, xmm0, mode_imm`
      mode 0 = round to nearest (round)
      mode 1 = round toward -inf (floor)
      mode 2 = round toward +inf (ceil)
      mode 3 = round toward zero (truncate)
    IF SSE4.1 not available (rare — only very old x86-64 CPUs):
      Use x87 `frndint` with the appropriate rounding mode set in the FPU CW.

  Float-to-string (Grisu2):
    The Grisu2 algorithm uses 64-bit integer arithmetic to produce the shortest
    decimal that round-trips. It runs in O(1) for 99.5% of doubles.
    Inputs: the float value, number of decimal places or "shortest".
    Output: a digit string and exponent.
    Implementation complexity: approximately 200 lines of NASM.
    Reference: Florian Loitsch, "Printing Floating-Point Numbers Quickly and
    Accurately with Integers", PLDI 2010.

─────────────────────────────────────────────────────────────────────────────
4.4  ALLOCATION PERFORMANCE — ARENA VS MALLOC
─────────────────────────────────────────────────────────────────────────────

The Rex arena allocator must be DRAMATICALLY faster than malloc. Benchmark target:
  10 million arena allocations: < 100 milliseconds (10 ns per allocation).
  10 million malloc calls: typically 2000–5000 milliseconds on Linux.
  Target speedup: 20x–50x.

This speedup comes from the arena's alloc_fn being literally 3 instructions:
  mov rax, [arena_bump]          ; load bump pointer
  add [arena_bump], size_aligned ; advance by aligned size
  ; rax now points to the allocated region

No lock acquisition. No free list search. No metadata header per allocation.
No thread-safety overhead (Rex is single-threaded).

The pool allocator:
  alloc_fn: `pop rax from free list` — essentially one `mov rax, [head]; mov r_temp, [rax]; mov [head], r_temp`.
  Five instructions for a pool allocation. Still 5x faster than malloc.

Benchmark:
  bench/alloc_bench.rex:
    use mm arena(512 * 1024 * 1024):  ; 512 MB slab
      int i = 0
      for i in 0..10000000:
        seq[int] tmp = 8               ; 8-element seq (pre-sized, no push)
    ; measure: time the above for loop
  Compare against bench/ref/alloc_bench.c which uses malloc(64)+free.

─────────────────────────────────────────────────────────────────────────────
4.5  HASH MAP PERFORMANCE — SURPASSING C HASH TABLES
─────────────────────────────────────────────────────────────────────────────

Rex dict must beat C hash table libraries (uthash, khash, glib GHashTable)
in lookup-heavy workloads. Strategies:

  1. INLINE KEY STORAGE: The key string is stored INLINE in the slot (up to
     15 bytes without extra allocation due to SSO). Lookup on short keys never
     follows a pointer to the key — it compares directly in the slot.
     For long keys (> 15 bytes): store a pointer and use the cached hash to
     filter before the pointer dereference.

  2. CACHE-LINE-ALIGNED SLOTS: Align the table start to 64 bytes. Align slot
     size to a power of 2. For dict[int] with short keys: slot is 32 bytes
     (key 15 bytes + 1 len byte + hash 8 bytes + value 8 bytes). Two slots
     fit in one cache line. A miss-free lookup touches at most 2 cache lines.

  3. ROBIN HOOD HASHING (alternative to tombstone linear probing):
     On insertion, if the new element's probe distance is greater than the
     element in the current slot, steal the slot and continue inserting the
     displaced element. This reduces average probe chain length dramatically.
     Average lookup: 1.1 probes at 75% load factor. Compare: linear probing
     without Robin Hood: 2.5 probes average at 75% load factor.
     Implement Robin Hood probing for dict to maximise lookup speed.

  4. SIMD PROBE: For tables with 16+ byte slot sizes: load 16 metadata bytes
     (hashed key fingerprints) with `movdqu`, broadcast the target fingerprint
     with `pshufb`, compare with `pcmpeqb` + `pmovmskb`. Check up to 16 slots
     in one comparison. This is the strategy used by Swiss Table (Abseil) and
     Rust's hashbrown.

─────────────────────────────────────────────────────────────────────────────
4.6  SORT PERFORMANCE — BEATING QSORT
─────────────────────────────────────────────────────────────────────────────

Rex's `seq.sort()` must beat C's `qsort()` by at least 10% on a random 1M
int64 array. This is achievable because:
  1. `qsort` uses an indirect comparator call (function pointer per compare).
     Rex's `sort()` on `seq[int]` knows the type at compile time and emits
     direct `cmp` instructions — no function call overhead per comparison.
  2. Rex's introsort uses median-of-three pivot selection (qsort often uses
     median-of-one or random pivot, which degrades on sorted/reversed inputs).
  3. Rex switches to insertion sort for N ≤ 16 (qsort implementations vary).

  Additional optimisation: pdqsort (Pattern-Defeating Quicksort) by Orson Peters.
  Pdqsort is superior to introsort on partially sorted, pipe-organ, or all-equal
  inputs. Consider implementing pdqsort instead of introsort.
  Key pdqsort improvements over introsort:
    - Detects "already sorted" and "reverse sorted" patterns in O(n) before
      partitioning, and handles them as O(n) special cases.
    - Uses block partitioning to improve branch prediction during the partition.
    - Scrambles the input deterministically when bad pivot quality is detected.

─────────────────────────────────────────────────────────────────────────────
4.7  STRING OPERATION PERFORMANCE
─────────────────────────────────────────────────────────────────────────────

Benchmark targets (all measured against gcc -O2 equivalent C programs):

  str.contains (10 KB haystack, 5-char needle):  Rex ≥ 3x faster than strstr
  str.lower / upper (10 KB string):               Rex ≥ 2x faster than tolower loop
  str.join (1000 strings, avg 10 bytes each):     Rex ≥ 2x faster than strcat loop
  str.split (10 KB string, 100 separators):       Rex ≥ 1.5x faster than strtok
  str concat (1000 short strings):                Rex ≥ 5x faster than strcat (arena)

  The key advantage is that Rex `str.join` allocates ONCE. A C program using
  strcat reallocs repeatedly. Rex's SSO means short strings never hit the heap.

4.8  BINARY SIZE MINIMISATION
─────────────────────────────────────────────────────────────────────────────

Rex binaries are small because they have no runtime:
  - No libc startup code (no _init, __libc_start_main, etc.)
  - No dynamic linker (statically linked against nothing)
  - No exception handling tables
  - No DWARF debug info in production builds (available with `rex build --debug`)
  - No padding between functions beyond alignment requirements

Target: "Hello, Rex!" binary ≤ 4 KB ELF64.
  ELF header: 64 bytes
  Program headers: 3 * 56 = 168 bytes (LOAD, LOAD, PT_GNU_STACK)
  .text section: ~50 bytes for the hello world code
  .rodata section: 14 bytes for "Hello, Rex!\n"
  Section headers: minimal
  Total: under 4 KB is achievable with careful layout.

Use `ld --strip-all` for production builds. Strips all symbols and debug info.
Use `objcopy --remove-section=.comment` to remove NASM version comments.
Do NOT use PHDR program header for the rex binary.

─────────────────────────────────────────────────────────────────────────────
4.9  STARTUP TIME MINIMISATION
─────────────────────────────────────────────────────────────────────────────

Rex's _start must do exactly:
  1. Save argc and argv for `os.args()` — two `mov` instructions.
  2. Call rex_init: CPUID check, fill dispatch table, set up default allocator.
     Target: < 10 microseconds for rex_init.
  3. Call the user's main protocol.
  4. Call sys_exit with the return value.

Total startup including kernel: < 50 microseconds.
Compare: Python starts in 50+ milliseconds. Node.js: 150+ milliseconds.
Rex: under 50 MICROSECONDS. This is a 1000x improvement over scripting languages.

rex_init must NOT:
  - Call any syscall other than mmap (for the default heap setup) and
    the CPUID instruction (not a syscall, just an instruction).
  - Run any protocol or library initialisation unless explicitly declared with
    `use mm static:` at module level.
  - Touch the file system.

§5  MEMORY SAFETY — COMPLETE REQUIREMENTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

─────────────────────────────────────────────────────────────────────────────
5.1  BOUNDS CHECKING — EVERY ACCESS, EVERY TIME
─────────────────────────────────────────────────────────────────────────────

Every `seq[i]`, `arr[i]`, `str[i]` index access in safe mode emits:
  ; bounds check for seq_name[i]
  mov rax, i_reg              ; or immediate value
  cmp rax, [seq_len_slot]     ; compare against length
  jge .bounds_error_NNN       ; >= length → out of bounds

  .bounds_error_NNN:
    ; (cold path — far from hot code)
    ; emit error: "rex: file.rex:LINE: bounds error: index VAL out of range [0..LEN]"
    ; format the error string at runtime using the actual values
    mov rdi, 2
    mov rsi, error_str_ptr
    mov rdx, error_str_len
    mov rax, 1
    syscall
    mov rax, 60
    mov rdi, 1
    syscall

Negative index handling BEFORE the bounds check:
  test rax, rax               ; is index negative?
  jns .bounds_check           ; no: go to positive bounds check
  add rax, [seq_len_slot]     ; add length (e.g., -1 + 5 = 4 = last element)
  test rax, rax               ; still negative after adding length?
  js .bounds_error_NNN        ; yes: was more negative than -len
  .bounds_check:
  cmp rax, [seq_len_slot]
  jge .bounds_error_NNN

The error path is marked `[section .cold]` so it does NOT pollute the
instruction cache. The check itself (test + jns + cmp + jge) is 4 instructions
and is predicted-not-taken (never fires in correct programs). Cost: ~1 clock
cycle overhead per access due to branch prediction.

BOUNDS CHECK ELIMINATION (Advanced):
  When the compiler can prove a bound statically, eliminate the check:
    - `for i in 0..seq.len():` — i is always valid, eliminate check on seq[i]
    - `if i >= 0 and i < seq.len():` before `seq[i]` — eliminate check
    - `arr[T, 5]` accessed with literal `arr[3]` — always valid, eliminate
  Track "definite bounds" in the semantic analysis pass. Emit no check when
  the proof holds. This recovers performance for index-heavy hot loops.

─────────────────────────────────────────────────────────────────────────────
5.2  NULL POINTER PROTECTION
─────────────────────────────────────────────────────────────────────────────

All heap collection pointers (seq, dict, str) are zero-initialised at declaration.
Before ANY method call on a collection, emit:
  test r_ptr, r_ptr           ; is the pointer null?
  jz .null_error_NNN          ; yes → error

.null_error_NNN:
  ; "rex: file.rex:LINE: null dereference: 'NAME' was never initialised"
  ; (cold path)

The compiler tracks which variables have been initialised in semantic analysis.
If a variable is proven-initialised (declaration with initialiser, or after any
assignment), skip the null check. The null check is only emitted when the
compiler cannot prove the variable is non-null.

─────────────────────────────────────────────────────────────────────────────
5.3  INTEGER OVERFLOW PROTECTION
─────────────────────────────────────────────────────────────────────────────

In safe mode, integer overflow is an error (not wraparound):
  add rax, rbx
  jo .overflow_error_NNN       ; jump on overflow flag set

.overflow_error_NNN:
  ; "rex: file.rex:LINE: arithmetic overflow: add overflowed (result wrapped)"
  ; Include the operand values in the message if they are in registers.

Overflow checks are emitted after: add, sub, imul, neg, shl (left shift by N).
Overflow check is NOT emitted after: and, or, xor, shr, sar (cannot overflow).
Overflow check for shl: `mov rcx, N; shl rax, N; jo .overflow_error` —
  left shift overflow means the high bits were non-zero before the shift.

In `#unsafe` blocks: no overflow checks. Two's-complement wraparound behaviour.
This allows Rex to be used for deliberate wraparound arithmetic (hash functions,
checksums, etc.) without the overhead of overflow checking.

─────────────────────────────────────────────────────────────────────────────
5.4  ESCAPE ANALYSIS — STACK ALLOCATION OF SHORT-LIVED OBJECTS
─────────────────────────────────────────────────────────────────────────────

A major performance optimisation: if the compiler can prove that a collection
never escapes the current scope (never stored in another collection, never
returned, never passed to a protocol that stores it), it may be allocated on
the stack instead of the heap.

Example:
  seq[int] tmp = [1, 2, 3]    ; tmp is only used below
  output(tmp.sum())
  ; tmp never escapes — allocate on stack

Stack allocation of seq[1,2,3]: emit `sub rsp, 40` (16-byte header + 24 bytes
data), fill inline. No heap touch. No allocation cost.

Escape analysis algorithm:
  For each heap collection declaration in a protocol:
    Walk all uses. If any of these are present, the object ESCAPES:
      - Passed to another protocol (it might store the pointer)
      - Stored in a seq or dict (the outer collection outlives the scope)
      - Returned from the protocol
      - Stored via a `:` write site to a higher-scope variable
    If NO escape use is found: mark the collection as stack-allocatable.
  Emit stack allocation for non-escaping collections.

This analysis can be performed conservatively (safe but may miss some cases)
without being a full alias analysis — which would require a full-program
data-flow pass. Conservative escape analysis is sufficient for the common case.

─────────────────────────────────────────────────────────────────────────────
5.5  WHOLE-PROGRAM OPTIMISATION MODE
─────────────────────────────────────────────────────────────────────────────

`rex build --wpo` enables whole-program optimisation:
  - All modules are compiled together in a single compilation unit.
  - Cross-module inlining: a `#inline` protocol defined in one module can be
    inlined at its call sites in another module.
  - Cross-module constant folding: if a protocol in module A is `#pure #total`
    and called with compile-time constants in module B, fold the call.
  - Cross-module escape analysis: if a seq created in module A is passed to a
    protocol in module B that is proven to not store it, the seq is not escaped.
  - Cross-module dead code elimination: protocols that are never called from
    any reachable code path are eliminated from the binary entirely.
  --wpo is slower to compile but produces the fastest binary.

§6  LANGUAGE SERVER PROTOCOL — FULL SPECIFICATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

An uneditable language is a dead language. The LSP is not a bonus feature —
it is a core deliverable. Without the LSP, Rex cannot be adopted.

Start the language server: `rex lsp`
Protocol: LSP 3.17 over JSON-RPC 2.0 over stdio.
Implementation language: write in C as a bootstrap (Rex can rewrite itself
once Rex is capable of compiling the LSP, which requires the io and json stdlib).

─────────────────────────────────────────────────────────────────────────────
6.1  SERVER LIFECYCLE
─────────────────────────────────────────────────────────────────────────────

  `initialize` request: respond with server capabilities.
  Capabilities to advertise:
    textDocumentSync: Incremental (send only changed ranges)
    completionProvider: { triggerCharacters: [".", "@", "{"] }
    hoverProvider: true
    definitionProvider: true
    signatureHelpProvider: { triggerCharacters: ["(", ","] }
    renameProvider: { prepareProvider: true }
    documentFormattingProvider: true
    diagnosticProvider: { interFileDependencies: false, workspaceDiagnostics: false }

  `initialized` notification: begin watching open files.
  `textDocument/didOpen`: add to open files set, trigger diagnostics.
  `textDocument/didChange`: apply incremental changes, trigger diagnostics.
  `textDocument/didClose`: remove from open files set.
  `shutdown`: graceful shutdown.
  `exit`: terminate process.

─────────────────────────────────────────────────────────────────────────────
6.2  DIAGNOSTICS — textDocument/publishDiagnostics
─────────────────────────────────────────────────────────────────────────────

  Architecture: on each file change, fork a child process that runs the Rex
  compiler's `rex check` pipeline on the file. The child writes JSON-formatted
  diagnostic output to stdout. The LSP server reads this and publishes.
  Using a child process ensures the LSP server never crashes due to a compiler
  bug — the child may crash, but the server survives.

  Debounce: 200ms after the last keystroke before triggering compilation.
  If a new change arrives during compilation: cancel the in-flight compilation,
  restart the 200ms timer. Use SIGTERM on the child process to cancel.

  Diagnostic format:
  {
    "range": {
      "start": {"line": 41, "character": 8},
      "end":   {"line": 41, "character": 15}
    },
    "severity": 1,          // 1=Error, 2=Warning, 3=Info, 4=Hint
    "source": "rex",
    "message": "type error: expected int but got float"
  }

  The compiler must emit diagnostics in this format when run with `rex check --json`.
  Every compiler error must have an exact character range (not just a line number).
  This requires the parser to track character offsets, not just line numbers.

─────────────────────────────────────────────────────────────────────────────
6.3  COMPLETION — textDocument/completion
─────────────────────────────────────────────────────────────────────────────

  The LSP must perform "lazy compilation" for completion: parse up to the cursor
  position, build a partial semantic model, and offer completions without
  requiring the file to be error-free.

  After `.` on a typed variable:
    Look up the variable's type. Return all methods for that type.
    Completion item format:
    {
      "label": "contains",
      "kind": 2,             // Method
      "detail": "(sub: str) -> bool",
      "documentation": "true if substring sub found anywhere",
      "insertText": "contains($1)",
      "insertTextFormat": 2  // Snippet
    }

  After `@`:
    Return all protocols defined in the current file and all imported modules.
    Format: `{ "label": "myProtocol", "kind": 3, "detail": "(int x) -> int" }`

  Inside `{` in a string:
    Return all variables in scope as completion items.
    Type: Variable (6), with detail showing the variable's type.

  Keyword completion: when no `.` or `@` context, return Rex keywords as
  items with kind=14 (Keyword).

  Filtering: the LSP client handles prefix filtering — return all items and let
  the client apply its own fuzzy matching. Do not pre-filter on the server side.

─────────────────────────────────────────────────────────────────────────────
6.4  HOVER — textDocument/hover
─────────────────────────────────────────────────────────────────────────────

  At any position in the document, identify what is under the cursor:
    - Variable: show `name: type`
    - Protocol call `@name(...)`: show full signature + doc comment (if `///` exists)
    - Method call `.method()`: show method signature + Notes from syn.md
    - Type keyword: show the one-line description from syn.md's Data Types table
    - String literal: show its length
    - Integer/float literal: show its type and value in decimal + hex

  Format: Markdown in the `contents.value` field. Use code fences for signatures.
  Response latency: < 50ms including the semantic lookup.

─────────────────────────────────────────────────────────────────────────────
6.5  GO-TO-DEFINITION — textDocument/definition
─────────────────────────────────────────────────────────────────────────────

  F12 on a protocol call `@foo(...)`: return the location of `prot foo(`.
  F12 on a variable: return the location of its declaration (type + name).
  F12 on a module-qualified call `@math.sqrt`: return the location in math.rex.
  F12 on a stdlib protocol: return a special location in a generated docs file
    (which the LSP creates on-demand: `~/.rex/docs/math.rex.html`).

  The location is a URI + range:
  {
    "uri": "file:///path/to/file.rex",
    "range": {
      "start": {"line": 10, "character": 0},
      "end":   {"line": 10, "character": 20}
    }
  }

─────────────────────────────────────────────────────────────────────────────
6.6  SIGNATURE HELP — textDocument/signatureHelp
─────────────────────────────────────────────────────────────────────────────

  Triggered when `(` is typed after a protocol or method name.
  Updated as the user types `,` to advance the active parameter.

  Response format:
  {
    "signatures": [{
      "label": "prot add(int a, int b) -> int",
      "parameters": [
        {"label": [9, 14]},   // "int a" character range
        {"label": [16, 21]}   // "int b" character range
      ],
      "documentation": "Doc comment if present"
    }],
    "activeSignature": 0,
    "activeParameter": 1     // 0-indexed, updated on each ","
  }

─────────────────────────────────────────────────────────────────────────────
6.7  RENAME — textDocument/rename
─────────────────────────────────────────────────────────────────────────────

  `textDocument/prepareRename`: validate that the identifier under cursor is
  renameable (not a keyword, not a built-in method name).
  Return the range of the identifier to be renamed.

  `textDocument/rename`: find ALL occurrences of the identifier in:
    1. The current file
    2. All files that import the module containing the definition
    3. All files imported by the current file
  Return a WorkspaceEdit with the set of text changes.

  Search strategy: re-parse each relevant file, collect all identifiers,
  filter by semantic equivalence (same binding, not just same text).

─────────────────────────────────────────────────────────────────────────────
6.8  FORMATTING — textDocument/formatting
─────────────────────────────────────────────────────────────────────────────

  Run `rex fmt --check` on the current file. If changes are needed, run
  `rex fmt --stdout` (format to stdout, leave file unchanged). Return the diff
  as a set of TextEdit objects (replace the entire file content with the
  formatted version — simplest correct approach).

  Format-on-save: most LSP clients support triggering format on save.
  The LSP should advertise `documentFormattingProvider: true` to enable this.

─────────────────────────────────────────────────────────────────────────────
6.9  SEMANTIC TOKENS — textDocument/semanticTokens
─────────────────────────────────────────────────────────────────────────────

  Provide semantic token information for rich syntax highlighting.
  This allows editors to highlight Rex-specific constructs that regex-based
  textmate grammars cannot express:
    - Protocol names in `@name()` calls: highlighted differently from variables
    - Decorator keywords `#memo`, `#hot`: highlighted as annotations
    - `:` at write sites: highlighted as a mutation marker
    - Type names in declarations: highlighted as types
    - `each`, `blast`, `pipe`: highlighted as loop keywords

  Token types to advertise:
    "namespace", "type", "class", "enum", "interface", "struct", "typeParameter",
    "parameter", "variable", "property", "enumMember", "event", "function",
    "method", "macro", "keyword", "modifier", "comment", "string", "number",
    "operator"

  Semantic tokens are computed in a full parse+semantic-analysis pass.
  Return as a compact encoded array per the LSP spec.

§7  FORMATTER — rex fmt — COMPLETE SPECIFICATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Rex has ONE canonical formatting style. No options. No `.rexfmt` config file.
The formatter is part of the language — consistent across all Rex codebases.

─────────────────────────────────────────────────────────────────────────────
7.1  WHITESPACE RULES
─────────────────────────────────────────────────────────────────────────────

  Indentation: 4 spaces. NEVER tabs. Mixed indent is always an error.
  Max line length: 100 characters. Lines longer than 100 are reformatted.
  Trailing whitespace: stripped from every line, no exceptions.
  Final newline: every file ends with exactly one newline character.
  Blank lines between top-level prot definitions: exactly one blank line.
  Blank lines between decorators and prot: NONE. `#memo` must be on the line
    immediately above `prot`, with no blank line between.
  Blank lines within a protocol body: at most one consecutive blank line.
  Leading blank lines in a file: removed.

─────────────────────────────────────────────────────────────────────────────
7.2  EXPRESSION FORMATTING
─────────────────────────────────────────────────────────────────────────────

  Binary operators: exactly one space on each side: `a + b`, `x * 2`, `n % m`.
  Comparison: `x == y`, `a != b`, `n <= 10`.
  Assignment: `:x = value` — one space after `:`, one space around `=`.
  Unary prefix: no space: `-x`, `~flags`, `not ready`.
  Method chains:
    - On one line if total line length ≤ 100 chars:
      `s.trim().lower().contains("hello")`
    - Multi-line if > 100 chars: each method on its own line, indented 4 spaces
      from the base expression:
      ```
      s
          .trim()
          .lower()
          .contains("hello")
      ```
  Parenthesised expressions: space after `(` and before `)` in multi-line
    expressions; no space for simple single-value `(x)` uses.

─────────────────────────────────────────────────────────────────────────────
7.3  COLLECTION LITERAL FORMATTING
─────────────────────────────────────────────────────────────────────────────

  Short seq literals (≤ 100 chars): `[1, 2, 3]` — space after `[`, before `]`
    only if the literal has elements.
  Long seq literals: break after each element with trailing commas:
    ```
    [
        1,
        2,
        3,
    ]
    ```
  Short dict: `{"key": val, "other": val2}` — space after `{`, before `}`.
  Long dict: same multi-line pattern as seq, each entry on its own line.
  Empty collections: `[]`, `{}`, `{}` — no spaces inside.

─────────────────────────────────────────────────────────────────────────────
7.4  CONTROL FLOW FORMATTING
─────────────────────────────────────────────────────────────────────────────

  `if / elif / else`: no space before `:`.
  `for / while / each / repeat`: no space before `:`.
  Body: always indented 4 spaces. No single-line bodies (even for `pass`).
  Blank line before `if` / `for` / `while` / `try` / `use mm` blocks when
    preceded by any statement (not at the start of a function).
  Blank line after the last statement in a block is removed.

─────────────────────────────────────────────────────────────────────────────
7.5  COMMENT FORMATTING
─────────────────────────────────────────────────────────────────────────────

  Inline comments: at least 2 spaces before `//`. Exactly one space after `//`.
    If the comment is pushed beyond column 100: move to the line above.
  Block comments `/* */`: preserve internal content but reindent the opening
    and closing delimiters to match the current indentation.
  Doc comments `///`: always on their own line. Exactly one space after `///`.
    Multiple `///` lines form a single doc block.

─────────────────────────────────────────────────────────────────────────────
7.6  IDEMPOTENCY GUARANTEE
─────────────────────────────────────────────────────────────────────────────

  `rex fmt` MUST be idempotent: running it twice produces the same output as
  running it once. This is verified by running `rex fmt | rex fmt` and checking
  that the output is identical. This check must be part of the formatter test suite.
  Every code example in syn.md must be in canonical formatter output — they are
  the ground truth for formatting style.

§8  TOOLCHAIN — INSTALLABILITY AND CLI
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Rex must be trivially installable on any x86-64 Linux system with zero
dependencies. A developer with a fresh Ubuntu 22.04 LTS machine should be
able to install Rex and compile their first program in under 2 minutes.

─────────────────────────────────────────────────────────────────────────────
8.1  INSTALLATION
─────────────────────────────────────────────────────────────────────────────

  One-line install (online):
    curl -fsSL https://rex-lang.dev/install.sh | sh

  The install script must:
    1. Detect OS: Linux only (print helpful error on macOS/Windows).
    2. Detect architecture: x86-64 only (print helpful error on ARM/RISC-V).
    3. Determine latest release version from GitHub releases API.
    4. Download: `rex-linux-x86_64-v5.0.tar.gz`
    5. Verify SHA256 checksum against the published checksum file.
       If mismatch: "error: download checksum mismatch. Aborting." and exit 1.
    6. Extract: the tarball contains exactly one file: `rex`.
    7. Install: copy to `~/.local/bin/rex` (no sudo required).
       If user wants /usr/local/bin: `curl ... | sudo sh` also works.
    8. Verify: run `rex --version` and print its output.
    9. Print: "\nRex V5.0 installed successfully.\nRun 'rex new myapp' to begin.\n"
    10. Add PATH hint if `~/.local/bin` is not in PATH.

  Manual install:
    Download `rex` binary. `chmod +x rex`. Copy to anywhere on PATH. Done.
    No other files required. No system configuration. No registry entries.

  Uninstall:
    `rm $(which rex)` — that is the entire uninstall.

  The `rex` binary is a statically linked ELF64 executable. `ldd rex` must
  print "not a dynamic executable". It must run on any Linux kernel >= 3.0
  (the mmap2 / mmap syscall interface was stable since before 3.0).

─────────────────────────────────────────────────────────────────────────────
8.2  CLI REFERENCE — COMPLETE
─────────────────────────────────────────────────────────────────────────────

  rex --version
    Print: "Rex V5.0 (NASM ELF64) built YYYY-MM-DD"
    Exit 0.

  rex --help
    Print a short, readable help page. Maximum 40 lines.
    List every subcommand with a one-sentence description.
    Exit 0.

  rex build [file.rex] [--release | --debug] [--wpo]
    Compile the given .rex file. Defaults to main.rex if not specified.
    Output binary: same name as input file without extension (or 'main').
    --release: strip symbols, apply all optimisations, smallest binary.
    --debug: include line number info, no stripping, emit .asm intermediate.
    --wpo: whole-program optimisation across all imports.
    Exit 0 on success, 1 on error (with error messages to stderr).
    On success: print "Built ./main (N bytes) in M.MMms"

  rex run [file.rex] [-- args...]
    Compile to a temp file and execute immediately. Pass args after `--`.
    Delete the temp binary after execution.
    The process's exit code is the Rex program's exit code.

  rex check [file.rex] [--json]
    Lex + parse + semantic analysis only. No NASM, no ld.
    Print all errors and warnings.
    --json: emit diagnostics as JSON array (for editor integration).
    Exit 0 if clean, 1 if errors found, 2 if warnings only.
    Must complete in < 100ms for a 1000-line file.

  rex fmt [file.rex | --check | --stdout]
    Format the file in-place using the canonical Rex style.
    --check: exit 0 if already formatted, 1 if changes would be made. No output.
    --stdout: write formatted output to stdout, do not modify the file.
    With no options: modify the file in-place.
    Useful in CI: `rex fmt --check && echo "style OK"`

  rex lsp
    Start the language server on stdin/stdout. Never exits until killed.
    Log to stderr (which editors redirect to a log file).

  rex new <project-name>
    Create directory <project-name>/ with:
      main.rex        — starter program: output("Hello, {project-name}!")
      tests/          — directory with test_main.rex
      bench/          — directory
      rex.toml        — project manifest
    Print: "Created project '<project-name>'. Run: cd <project-name> && rex run"

  rex test [file.rex | --all]
    Compile and run test files. Each test file is in tests/.
    A test file that exits 0 → PASS. Exits non-zero or crashes → FAIL.
    `assert` inside test files: failure prints the assertion and exits 1.
    --all: run every .rex file in tests/ recursively.
    Print: "37/37 tests passed" or list of failures with line numbers.
    Exit 0 if all pass, 1 if any fail.

  rex bench [--all | file.rex]
    Run benchmark files in bench/. Each benchmark must print its result.
    --all: run every benchmark.
    After each benchmark: also run the matching bench/ref/<name>.c (compiled
    with gcc -O2) and print the ratio: "Rex: 1.2x faster than C"
    Exit 0 always.

  rex doc [file.rex | --all]
    Generate HTML documentation from `///` doc comments.
    Output: ./doc/index.html
    Each `prot` with `///` comments gets its own anchor and section.
    The output is static HTML — no JavaScript required.

  rex asm [file.rex]
    Compile and emit the intermediate NASM file without assembling it.
    Output: <name>.asm in the current directory.
    Useful for inspecting the generated assembly.

§13  PHILOSOPHY AND LONG-TERM VISION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Rex exists because the gap between "fast" and "safe" languages has been accepted
as a law of nature. It is not. The gap exists because fast languages (C, C++) were
designed before we understood type systems, and because safe languages (Rust,
Go, Haskell) were designed by people who mistrusted hardware.

Rex trusts the hardware. Rex trusts the programmer. Rex trusts assembly code.
But Rex also believes that safety is not the enemy of speed — it is merely
an engineering problem with a known solution: know what your program is doing
at every moment, at the compiler level, and check only what cannot be proven safe.

─────────────────────────────────────────────────────────────────────────────
13.1  WHY NASM AS THE TARGET
─────────────────────────────────────────────────────────────────────────────

NASM is the optimal output format for a language with Rex's goals because:
  1. NASM is readable by any x86-64 programmer. The compiler's output is legible.
  2. NASM has no hidden transformations. What you write is what gets assembled.
  3. NASM lets the Rex compiler own every instruction. No LLVM optimiser between
     the compiler and the metal means no surprises.
  4. NASM's flat assembly model is a perfect match for Rex's own flat type system.
  5. NASM-assembled ELF binaries are small. The assembler adds no padding,
     no metadata, no exception tables unless explicitly requested.

The tradeoff: Rex's compiler must implement all optimisations itself. This is
not a disadvantage — it is a design goal. Every optimisation in Rex is
intentional, documented, and testable. There are no "mystery speedups" from
LLVM that disappear after an LLVM update.

─────────────────────────────────────────────────────────────────────────────
13.2  THE `:` SIGIL — WHY IT MATTERS
─────────────────────────────────────────────────────────────────────────────

The `:` write-site sigil is Rex's single most important design decision.
It makes mutations visible at a glance without requiring an IDE, without a
linter, without any tooling. A programmer reading Rex code can scan it with
their eyes and immediately see every place where state changes.

This has a real performance implication: the compiler knows, statically, which
variables are never mutated (those with no `:` write sites). Those variables
are compile-time constants, eligible for constant folding and inline substitution.
A variable declared `int x = 42` with no `:x = ...` anywhere in scope is
ALWAYS 42 — the compiler replaces every read of x with the literal 42.

This is more powerful than `const` in C because it applies to ALL declarations
by default — the programmer opts INTO mutability, not out of it.

─────────────────────────────────────────────────────────────────────────────
13.3  THE ALLOCATOR CONTEXT — WHY IT IS THE RIGHT ABSTRACTION
─────────────────────────────────────────────────────────────────────────────

Rex's `use mm arena:` is not a feature — it is a philosophy. The philosophy is:
the programmer chooses the memory strategy at the call site, not at the definition
site. A protocol that allocates memory (`.push()`, `.split()`, `.map()`) does not
need to know or care what allocator it is using. The caller decides.

This is the inversion of Zig's explicit allocator threading, and it is better.
In Zig, every function that allocates must accept an allocator parameter, and
every caller must pass one. This is correct but verbose. In Rex, the allocator
flows through the call stack invisibly, stored at a fixed address, with zero
argument overhead and zero calling convention changes.

The result: Rex programs can switch from heap to arena to pool to custom
allocator with a single `use mm` wrapper, without changing any protocol
signatures, without refactoring any inner code. This is the fastest possible
path to production performance for allocation-heavy programs.

─────────────────────────────────────────────────────────────────────────────
13.4  WHY REX WILL SURPASS C
─────────────────────────────────────────────────────────────────────────────

C is held back by its history. C was designed in 1972 for a PDP-11 with 64 KB
of RAM. Its memory model (everything is a pointer to an untyped byte sequence),
its string model (null-terminated, no length prefix, no bounds checks), and its
allocation model (malloc/free, global heap, no context) were reasonable choices
for 1972. They are poor choices for 2025.

Rex chooses differently:
  - Strings have length prefixes. No strlen(). No null-termination bugs.
  - Collections know their type. No void*. No size_t arithmetic.
  - Allocators are contextual. No global lock contention.
  - Bounds are checked by default. No buffer overflows.
  - Overflow is detected by default. No silent wraparound.

Each of these choices ADDS safety. Most of them also IMPROVE PERFORMANCE:
  - Length-prefix strings: O(1) length (not O(n) strlen). O(1) capacity check.
  - Typed collections: the compiler knows sizeof(T), so pointer arithmetic is
    always correct and the loop body never needs to compute element size at runtime.
  - Contextual allocators: arena allocation is 10x faster than malloc. A Rex
    program that uses arenas correctly is faster than any C program using malloc.
  - Bounds-check elimination: Rex can prove more bounds safe than C can, because
    Rex knows more about the program. The result is that many bounds checks are
    eliminated at compile time — and the ones that remain are correctly predicted.

Rex is not "as fast as C with a safety tax". Rex is "faster than C because safe
design enables better performance."

─────────────────────────────────────────────────────────────────────────────
13.5  THE VISION FOR V6.0 AND BEYOND
─────────────────────────────────────────────────────────────────────────────

V5.0 is the complete, stable, production-quality Rex. V6.0 will add:
  - Self-compilation: the Rex compiler becomes a Rex program.
  - Parametric types: `seq[T]` where T is a type parameter to protocols.
  - Traits/protocols as types: pass a protocol to another protocol.
  - AVX-512 support on Zen 4 and Intel Sapphire Rapids.
  - RISC-V backend: Rex on embedded systems.
  - WebAssembly backend: Rex in the browser via WASM.
  - Package registry: `rex add <package>` downloads from rex-lang.dev/pkg.
  - Mutual recursion TCO: trampoline-based tail call for mutual recursion.
  - Compile-time execution: evaluate `#pure #total` protocols at compile time.
  - Ownership/borrow checker (optional): verify `own/move` correctness.

But all of that is V6.0. V5.0 must be perfect first. Build V5.0 completely,
correctly, and measurably fast before thinking about what comes next.
