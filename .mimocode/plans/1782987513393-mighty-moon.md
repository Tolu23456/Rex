# Register Pinning Logic: Pin Accumulator to r14 in Loop Bodies

## Problem
The multi-register cache infrastructure (`ra_var[4]`, `ra_reg[4]`, `ra_count`) and the load/store paths that check it are fully implemented but **never populated**. `codegen_while_pin_setup` already emits `push r14` + 8-NOP placeholder, but the NOPs are never patched and `pop r14` is never emitted. Accumulator variables (like `total` in `total = total + i`) are still accessed from memory on every iteration.

## Goal
Pin the accumulator variable (detected by O-G fold) to r14 during loop bodies, eliminating memory traffic. Each load/store of the pinned variable uses 3-byte register ops instead of 8-byte memory ops.

## Key Constraint: O-G Fold Conflict
The O-G fold in `codegen_emit_store_rax_to_var` detects patterns like `mov rax,[addr](8) + OP rax,reg(3)`. If the accumulator is pinned to r14 BEFORE the body, the load becomes `mov rax, r14` (3 bytes) which breaks the O-G pattern detection. **Solution: Deferred patching** — emit NOP placeholder at loop start, patch after O-G fires.

## Current State (what's already built)
- **BSS**: `ra_var[4]`, `ra_reg[4]`, `ra_count`, `ra_hoist_slot_pos` — all declared (line 219-224)
- **`codegen_while_pin_setup`** (line 5953): Already emits `push r14` (41 56) + 8-NOPs + records `ra_hoist_slot_pos` + emits `mov r15, [counter_addr]`
- **Load/store paths**: `codegen_emit_mov_rax_var` (line 1294) and `codegen_emit_store_rax_to_var` (line 1535) already scan `ra_var[0..3]` and emit 3-byte register ops for r14/r13/r12
- **Missing**: NOP patching in while_end, r14 flush/pop in while_end, push r14 + NOPs in for_start, NOP patching + flush/pop in for_end

## x86-64 Encoding Reference

| Instruction | Encoding | Size |
|---|---|---|
| push r14 | `41 56` | 2 bytes |
| pop r14 | `41 5E` | 2 bytes |
| mov r14, [abs32] | `4C 8B 34 25 addr32` | 8 bytes |
| mov [abs32], r14 | `4C 89 34 25 addr32` | 8 bytes |
| NOP | `90` | 1 byte |
